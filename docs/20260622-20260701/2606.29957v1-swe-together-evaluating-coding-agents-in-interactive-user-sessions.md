## 摘要
大多数编码智能体基准测试是静态的：智能体一开始就收到完整的任务描述，并且仅根据其最终代码进行评判。真实的编码辅助是交互式的，用户会在多轮交互中澄清目标、添加约束并纠正错误。我们提出了SWE-Together，这是一个从真实用户-智能体编码会话中重建的多轮基准测试。为了使真实交互可验证，我们从11,260个记录的会话中精选了109个仓库级任务，选择了具有可恢复仓库状态、清晰用户目标和可观察结果的会话。为了在不同智能体之间回放这些交互，我们构建了一个基于LLM的反应式用户模拟器，该模拟器保留了原始用户的意图，并在编码智能体的进展需要时提供反馈。为了评估智能体作为协作者的表现，我们同时衡量了最终仓库的正确性以及交互过程中所需的纠正性反馈轮次数。对前沿编码智能体的实验表明，更强的智能体通常能达到更高的最终成功率，同时需要更少的干预，这表明用户体验得到了改善。

## Abstract
Most coding-agent benchmarks are static: an agent receives a complete task description up front and is judged only by its final code. Real coding assistance is interactive, with users clarifying goals, adding constraints, and correcting mistakes over multiple turns. We introduce SWE-Together, a multi-turn benchmark reconstructed from real user-agent coding sessions. To make real interactions verifiable, we curate 109 repository-level tasks from 11,260 recorded sessions, selecting sessions with recoverable repository states, clear user goals, and observable outcomes. To replay these interactions across agents, we build a reactive LLM-based user simulator that preserves the original users' intents and provides feedback when the coding agent's progress requires it. To evaluate agents as collaborators, we measure both final repository correctness and the number of corrective feedback turns required during the interaction. Experiments with frontier coding agents show that stronger agents generally achieve higher final success rates while requiring fewer interventions, suggesting an improved user experience.

# SWE-Together：在交互式用户会话中评估编码智能体

## 1. 核心问题与整体含义（研究动机与背景）

- **现有编码智能体基准的缺陷**：主流基准（如SWE-Bench、Terminal-Bench）采用**静态协议**——任务描述一次性给出，仅通过最终代码和测试用例评分。这忽略了真实编码辅助的本质是**交互式**的：用户会逐步澄清目标、添加约束、纠正错误，信息分布在多轮对话中。
- **评估维度的缺失**：现有评估只关注最终任务成功率，不衡量用户付出的交互成本（例如需要多少纠正性反馈）。两个模型可能最终得分相同，但一个需要大量用户干预，另一个仅需粗略初始指令即可成功。
- **核心挑战**：构建既能保留真实用户交互，又能保证可验证性的评估。原始会话日志不一定可复现（仓库状态不可恢复、用户目标不明确、缺少可观测结果），且不能直接回放给新智能体（后续用户回合需依赖被评估智能体的轨迹，同时保持原始意图不变）。
- **论文目标**：提出 **SWE-Together**——一个从真实用户-智能体多轮编码会话重建的交互式基准，同时评估**最终任务正确性**和**交互过程中用户引导量**，更贴近真实用户体验。

## 2. 方法论：核心思想、关键技术细节

### 2.1 核心思想
- 将真实的、多轮的用户-智能体对话日志转化为可重放的、沙盒化的仓库级任务。
- 构建一个**锚定、状态条件化的LLM用户模拟器**，在保持原始用户意图和干预顺序的同时，根据被评估智能体的轨迹自适应释放反馈。
- 采用**联合评估协议**：用冻结的、与实现无关的评分标准（rubric）评估最终仓库状态；同时通过**User Correction（用户修正）** 和**Intent Coverage（意图覆盖率）** 刻画交互轨迹。

### 2.2 会话到任务的构建（Session-to-Task Construction）
- **数据源**：来自四个开源上游数据集：DataClaw（2,228条）、Pi-staging（2,397条）、Hyperswitch（784条）、SWE-chat（5,851条），总计11,260条原始会话。
- **三阶段过滤**：
  - **阶段1（确定性过滤）**：基于规则筛选——必须有多轮真实用户消息、有智能体代码编辑动作、能识别工作仓库。过滤掉只有讨论或只读探索的会话；排除最终变更主要由人类撰写的会话。
  - **阶段2（可行性筛选）**：LLM裁判判断会话中的编码工作能否重构为本地可执行的、自包含的任务。拒绝依赖外部状态（如PR管理、部署、凭证、实时服务）的会话。
  - **阶段3（任务构建）**：沙盒中由任务生成智能体完成：克隆仓库、固定提交ID、确定设置和测试命令、编写任务工件。最终输出包括初始用户指令、固定环境、确定性验证器、用户模拟器提示。
- **最终规模**：109个任务（转换率0.97%），保证高精度。

### 2.3 用户模拟器（User Simulator）
- **设计原则**：干预是**轨迹条件化**而非固定调度；**锚定于原始会话**而非通用。
- **决策方式**：在每个智能体回合后，模拟器根据以下信息做出一次性决策：
  - 智能体的最新动作、回复、仓库变化、已用时间
  - 模拟器自身的历史决策
  - 从原始会话重构的**会话分析**（包括用户目标、约束、干预条件）
- **动作空间**：`no-op`（默认，保持沉默）、`question`（提问澄清）、`redirect`（重定向）、`new-requirement`（新需求）、`check-external`（检查外部工件）。
- **避免失败模式**：固定回放可能在智能体走不同路径时时机不当；通用模拟可能偏离原始任务。

### 2.4 评估方法

#### 2.4.1 任务正确性（Task Correctness）
- 采用**行为完整性**评分，而非与参考补丁的相似性。
- 使用**智能体Rubric裁判**（Agentic Rubric Judge），分两阶段：
  - **阶段1**：为每个任务离线推导加权评分标准（可参考原始参考补丁，但目标是行为级的）。
  - **阶段2**：对每个候选仓库状态应用相同标准，每个目标返回是否达成及证据。
- 最终得分：`score = round(∑_g w_g * I[g met], 2)`，权重归一化，支持部分分数。
- 避免单纯执行测试的缺陷（窄测试可能拒绝功能正确解，宽测试可能要求从未请求的行为）。

#### 2.4.2 用户模拟行为诊断
- **Intent Coverage（意图覆盖率）**：衡量模拟器忠实度。先分解原始会话的原子意图，然后在回放中匹配模拟器消息，计算加权意图召回率（0.70权重）和范围精确率（0.30权重）。得分稳定说明跨智能体可比较。
- **User Correction（用户修正）**：衡量智能体引起的纠正性引导。对模拟器每条消息进行多标签分类：
  - **纠正层**：`correction`（显式指出错误/不完整，权重1.0）、`nudge`（暗示怀疑/鼓励重新考虑，权重0.2）
  - **非纠正询问层**：request、question、verification（不计入）
  - **其他**：workflow、approval、context（不计入）
  - 公式：`UserCorrection = N_correction + 0.2 * N_nudge`；先求每个任务内多个复现的平均，再跨任务平均。

### 3.2 评估指标
- **pass@1**：边缘单次运行成功率。
- **SSR（Stable Solve Rate）**：先对任务内连续评分平均，再阈值化，衡量平均可靠性。
- **pass^2**：两次复现均超过阈值才记成功，最严格。
- **MeanJudge**：平均连续评分。
- **User Correction**：纠正性引导量（越低越好）。
- **输出+推理Token数**和**墙钟时间**。

## 6. 主要结论与发现

1. **性能排序**：Claude Opus 4.8在所有正确性指标上领先（pass@1=63%, SSR=59%, pass^2=52%, MeanJudge=0.801），且User Correction最低（1.38）。GPT-5.5排名第二（pass@1=58%, MeanJudge=0.763）。MiniMax-2.7最弱且需最多纠正。
2. **User Correction与能力强负相关**：User Correction与pass@1（Pearson -0.92）、SSR（-0.84）、MeanJudge（-0.93）强负相关。**更强的模型需要更少的用户干预**，验证了该指标能反映用户体验。
3. **效率差异**：GPT-5.5在Token和墙钟上最高效（29.9k tokens, 10.7 min），Claude Opus 4.8消耗Token最多（74.0k）但墙钟适中（23.3 min）。能力与成本并非单调相关。
4. **模拟器质量**：人机图灵测试中，模拟器轨迹被误认为是真实用户的概率为46%（95% CI [40.5%, 51.6%]），包含50%，说明人类无法可靠区分，模拟器逼真。
5. **Intent Coverage稳定**：6/7模型得分在0.70-0.72之间，说明模拟器忠实且跨模型可比。GPT-5.5略低（0.68），原因可能是更强智能体自己解决交互，减少了模拟器自然开口机会。
6. **参考补丁基线**：在93个有参考补丁的任务上MeanJudge=0.90，pass=78%（τ=0.85），说明任务具有挑战性且评分合理。

## 7. 优点

1. **真实数据驱动**：从11,260条真实用户会话中筛选，而非人工构造，更贴近真实编码辅助场景。
2. **交互式评估新维度**：不仅衡量最终结果，还引入User Correction反映用户交互负担，弥补了静态基准的不足。
3. **可复现且公平**：三阶段任务构建保证状态可恢复、环境固定（沙盒）；评分标准固定，避免偏差；模拟器行为一致性验证确保跨模型可比。
4. **模拟器设计创新**：轨迹条件化+锚定原始会话的决策策略，既适应不同智能体路径，又保持原始意图不变。图灵测试验证了拟人度。
5. **评估框架完善**：同时提供阈值指标（pass@1, SSR, pass^2）和连续指标（MeanJudge），以及交互诊断（User Correction, Intent Coverage），多维度刻画模型表现。
6. **开源与透明**：代码和网站已公开（GitHub: Togetherbench/SWE-Together）。

## 8. 不足与局限

1. **模拟器的限制**：无法在智能体执行过程中打断，不能直接编辑文件，仅依赖文本轨迹和工具输出，无法获取视觉信息。不适合需要视觉反馈或需用户直接操作的任务。
2. **任务覆盖范围窄**：仅109个任务，且集中于可测量成果（如提交补丁）。对模糊、开放式的任务覆盖不足，难以量化定性用户行为。
3. **数据源偏差**：上游数据集可能来自特定用户群体和编码助手（如Claude、GPT），可能无法代表所有真实世界编码交互。
4. **缺少消融与对比实验**：未与其他模拟器方法比较，未分析不同干预策略的影响，未评估不同复现次数对指标稳定性的影响。
5. **算力与成本未披露**：缺少训练或推理时使用的具体GPU型号或数量，不利于复现与资源评估。
6. **参考补丁基线不完美**：参考补丁平均分0.90而非1.0，部分因为过程要求无法在最终补丁中体现，导致所有智能体均在这些目标上扣分。虽然公平，但降低了绝对分数的可解释性。
7. **框架泛化性未验证**：仅使用`opencode`一个框架，未测试其他框架（如CodingAgent、AgentCoder）下的表现，可能夸大框架耦合影响。
