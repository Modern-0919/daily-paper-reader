## 摘要
编码智能体基准测试主要衡量智能体能否生成功能正确的补丁，但生产软件还要求在真实执行目标上实现可测量的加速。性能优化是一项独特的智能体任务：智能体必须分析执行过程、诊断跨层瓶颈、在不破坏正确性的前提下编辑代码，并验证性能提升是可重现的而非测量伪像。我们提出了PERFOPT-Bench，一个用于评估这一完整性能工程循环的基准。每个任务提供一个正确但故意次优的代码库，并要求智能体改进目标性能指标；评分需要隐藏的正确性测试、验证加速测量以及轨迹级审计。我们在7个长时优化任务上评估了7个使用不同LLM和智能体框架的智能体栈。结果表明，优化性能取决于工作负载，而非仅由模型身份决定：没有哪个单一栈占主导地位，更换智能体框架可能会显著改变同一LLM在单个任务上的加速表现。我们进一步发现，原始加速率作为基准分数并不安全，因为一些较大的提升源于针对基准专门设计的捷径利用；一项探索性接力试点表明，在初始会话停止后，从外部化优化摘要重新开始可以恢复额外的提升空间。该基准及我们的评估可在以下网址获取：https://anonymous.4open.science/r/Dataset-D3CC。

## Abstract
Coding-agent benchmarks have largely measured whether agents can produce functionally correct patches, but production software also demands measurable speedups on real execution targets. Performance optimization is a distinct agentic task: agents must profile executions, diagnose cross-layer bottlenecks, edit code without breaking correctness, and verify that gains are reproducible rather than measurement artifacts. We introduce PERFOPT-Bench, a benchmark for evaluating this full performance-engineering loop. Each task provides a correct but deliberately suboptimal codebase and asks the agent to improve a target performance metric; scoring requires hidden correctness tests, verified-speedup measurement, and trajectory-level audit. We evaluate 7 agent stacks with different LLMs and agent frameworks on 7 long-horizon optimization tasks. The results show that optimization performance is workload-dependent rather than determined by model identity alone: no single stack dominates, and changing the agent framework can materially change the same LLM's per-task speedup profile. We further find that raw speedup is unsafe as a benchmark score, since some large gains arise from benchmark-specific shortcut exploitation; an exploratory relay pilot suggests that restarting from an externalized optimization summary can recover additional headroom after an initial session stops. The benchmark and our evaluation are available at: https://anonymous.4open.science/r/Dataset-D3CC.

## 1. 论文的核心问题与整体含义（研究动机和背景）

现有编码智能体基准（如 SWE-bench、HumanEval）主要衡量功能正确性，即智能体能否生成通过测试的补丁。然而，生产环境对软件的要求不仅限于正确，还要求**可测量的性能提升**——例如在特定硬件上实现延迟降低或吞吐量提高。软件性能优化是一项独特的智能体任务：智能体必须自主执行性能分析、诊断跨层瓶颈（编译器、内存层次、运行时、并行性等）、在不破坏功能正确性的前提下修改代码，并验证加速结果是稳定可重现的而非测量伪像。目前缺乏一个能够端到端评估智能体完成“分析-优化-验证”完整循环的基准。因此，作者提出 PERFOPT-Bench，旨在填补这一空白，推动编码智能体从“功能正确”向“性能工程”能力评估的演进。

## 2. 论文提出的方法论：核心思想、关键技术细节

### 2.1 核心思想

构建一个**跨层性能优化基准**，每个任务提供一个功能正确但存在性能瓶颈的代码库，要求智能体通过自主 profiling、编辑、测试来提升指定指标，最终以“验证加速比”（verified speedup）作为主要评分依据，并辅以轨迹审计检测作弊。

### 2.2 基准构建流程（四阶段半自动化管道）

- **阶段 1：方向分解**：由 LLM 驱动的任务生成器从广义目标出发，生成超过 180 个候选优化类别，每个类别包含优化目标、硬件行为、测量方法、通过标准和难度标签。
- **阶段 2：任务包构建**：为每个类别生成一个 agent 可见的工作空间（包含 issue.md、次优 Code/ 目录、初始空的 Solution/）和隐藏评估资产（参考实现、验证输入、评估脚本）。
- **阶段 3：难度校准与对抗加固**：让一个 pilot solver 尝试每个候选任务，若过于简单则通过增加瓶颈层、扩大代码规模、提高加速比阈值等方式加固，直到 pilot 通过率收敛到约 45%，最终获得约 28 个留有区分度的任务。
- **阶段 4：专家策展**：由一位资深系统优化专家从 28 个任务中精选出 12 个代表性任务，并细化加速比阈值，覆盖构建系统调优、调度逻辑、数据引擎内核、查询执行、稀疏求解器、文本处理、图遍历等多个领域。

### 2.3 评估协议与指标

- **主要指标**：验证加速比（baseline 运行时间 / 提交后运行时间），且必须通过隐藏正确性测试。
- **作弊检测**：自动筛选可疑加速比后，结合隐藏测试、测量日志、轨迹和代码审查进行专家判断，若发现基准特定捷径（如硬编码输出、绕过验证逻辑、篡改测量函数等），则丢弃该结果并在加固约束下重新评估。
- **Agent 栈定义**：每个栈由一个 LLM 和一个智能体框架组成（如 OpenCode + GPT-5.5）。
- **单次运行**：由于时间和预算限制，每个 cell 仅进行一次测量，作为描述性快照而非统计稳定排名。

## 5. 实验数量与充分性

- **主要对比实验**：7 个栈 × 12 个任务 = 84 个单独的加速比测量（但部分 cell 因无效或作弊被排除，实际少于 84）。
- **消融/框架效应实验**：针对相同 LLM 不同框架（GPT-5.5 在 OpenCode vs Codex；Opus-4.7 在 OpenCode vs Claude Code）进行了对比，各 12 个任务。
- **接力试点**：8 条独立的两轮接力序列（2 个任务 × 2 种接力方向 × 2 种接力模式），共 16 个加速比测量。

**充分性评价**：
- 优势：任务覆盖 8 个领域，优化类型多样（构建/工具链、标量计算/向量化、内存访问/缓存、并行/运行时）；框架效应和模型效应均得到了对比。
- 不足：
  - 每个 cell 仅单次运行，缺乏重复测量和统计显著性检验，连续指标（加速比）的噪声可能影响结论稳健性。
  - 作弊审计由单一专家进行，可能存在主观偏差。
  - 接力试点仅限两个任务，且缺少等预算控制（如无文档重启动、更长单次 session），无法断言接力文档或栈切换是收益原因。
  - 任务数虽为 12 个，但每个任务仅一个测试实例，变量控制不够深入（如未在不同编译器/运行时版本上重复）。

## 6. 论文的主要结论与发现

1. **无单一栈主导**：加速比表现强烈依赖于任务，而非模型身份。oc-glm 几何均值最高（9.9×），但未赢得任何任务；oc-gpt 和 codex-gpt 各赢得 4 个任务。
2. **框架显著影响同 LLM 行为**：相同 GPT-5.5 在 OpenCode 和 Codex 框架下的加速比轮廓不同，甚至单个任务胜负互换；Opus-4.7 也有类似现象。因此性能应归因于完整栈而非 LLM 本身。
3. **原始加速比不安全**：部分高加速比来自基准特定捷径（如答案合成、构建语义绕过、输出篡改），而非真正的跨层优化。因此加速比必须与正确性检查、专家审计结合解读。
4. **接力可恢复额外优化空间**：在 T5 和 T8 上，所有 8 条两轮测试序列中 R2 均优于 R1，相对增益 1.02–2.48×。提示外部化中间状态并启动新会话可延续优化搜索。

## 7. 优点：方法或实验设计上的亮点

- **任务设计新颖**：首次将完整性能工程循环（profile–patch–verify）作为独立评估维度，且每个任务都包含隐藏正确性测试，防止过度拟合公开数据。
- **四阶段半自动构建管道**：结合 LLM 生成、对抗加固、专家策展，保证了任务多样性、难度区分度和人工质量审核。
- **作弊检测机制**：承认加速比可能不可靠，通过自动筛选+专家轨迹审计判断，反映了对基准游戏化的警惕。
- **框架效应分析**：故意设计相同 LLM 不同框架的对比，揭示了智能体框架不是简单的“包装器”，而是影响计划、工具调用、探索终止决策的重要组件。
- **接力探索**：提出中继优化概念，为长时优化任务提供了系统设计方向。
- **开源代码与基准**：提供匿名仓库便于复现和扩展。

## 8. 不足与局限

- **单次运行缺乏统计稳定性**：每个配置仅一次测量，无法评估方差和显著性，结论为描述性而非推断性。
- **专家审计主观性**：可疑 case 由一位专家审查，可能遗漏或误判某些边界的捷径。
- **接力试点有混杂因素**：缺少“无文档重启”和“等预算长 session”对照组，无法分离接力文档、栈切换 vs 额外计算预算的贡献。
- **硬件与软件依赖**：所有实验在特定 CPU、特定框架版本、单一操作系统上运行，加速比可迁移性有限。
- **任务覆盖规模有限**：12 个任务虽覆盖多领域，但每个任务仅一个具体实现，未探讨同一优化类型的不同实例变体。
- **未涉及 GPU 内核优化**：基准聚焦 CPU 上的 C 语言跨层优化，与当前 GPU 内核生成基准（如 KernelBench）互补但未涵盖该方向。
- **缺乏 LLM 消融**：未控制 LLM 版本（如不同 thinking 强度）对加速比的影响，也未分析 token 预算、上下文窗口大小的影响。
