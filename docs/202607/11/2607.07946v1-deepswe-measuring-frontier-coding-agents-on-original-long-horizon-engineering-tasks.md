## 摘要
DeepSWE是一个包含113个原创、长周期软件工程任务的基准测试，用于评估编码智能体。大多数公开的智能体编码基准测试沿袭SWE-bench，从公共GitHub仓库中挖掘合并修复，这带来了两个问题：修复及其讨论可能在预训练期间已被见过，因此高分可能反映记忆而非问题解决能力；此外，每个任务由其合并修复附带的测试进行评分，这些测试是为确认某个特定修复而编写的，而非为任意解决方案评分，因此它们可能使正确的替代方案失败或使不完整的方案通过。DeepSWE避免了这两个问题。其任务从91个活跃的开源仓库和五种语言中从头编写，且从不向上游贡献，因此其参考解决方案不进入模型训练所抓取的公共记录；每个任务由手工编写的验证器评分，该验证器检查请求的功能并接受任何提供该功能的实现。当独立的LLM裁判重新评审已评分的运行结果时，其与DeepSWE验证器的不一致频率比与SWE-Bench Pro继承测试的不一致频率低一个数量级（1.4%对比32.4%）。尽管其提示长度约为SWE-Bench Pro的一半，但DeepSWE的提示所描述的任务，其参考解决方案涉及的代码量多出5.5倍，并且该基准测试在更宽的分数区间上区分前沿智能体，而相比之下它们在其他排行榜上则聚集在一起。我们发布了该基准测试、其验证器以及完整的评估轨迹记录。

## Abstract
DeepSWE is a benchmark of 113 original, long-horizon software engineering tasks for evaluating coding agents. Most public agentic coding benchmarks follow SWE-bench in mining merged fixes from public GitHub repositories, which creates two problems: the fixes and their discussion were likely seen during pretraining, so a high score can reflect recall rather than problem-solving; and each task is graded by the tests that shipped with its merged fix, which were written to confirm one specific fix rather than grade an arbitrary solution, so they can fail a correct alternative or pass an incomplete one. DeepSWE avoids both. Its tasks are written from scratch across 91 active open-source repositories and five languages and are never contributed back upstream, so their reference solutions stay out of the public record that model training scrapes; and each task is graded by a hand-written verifier that checks the requested functionality and accepts any implementation that provides it. When an independent LLM judge re-reviews graded runs, it disagrees with DeepSWE's verifier about an order of magnitude less often than with SWE-Bench Pro's inherited tests (1.4% versus 32.4%). Despite being about half the length of SWE-Bench Pro's prompts, DeepSWE's prompts describe tasks whose reference solutions touch 5.5x more code, and the benchmark separates frontier agents across a wider score band than the leaderboards on which they otherwise cluster. We release the benchmark, its verifiers, and the full record of evaluation trajectories.

3. **任务难度更高**：尽管提示长度仅为 SWE-Bench Pro 的一半，但参考解决方案代码量多 5.5 倍，文件数多约 1.5 倍，表明任务需要更多探索和实现工作。
4. **模型表现排名**：

   - 中段：Claude Sonnet 4.6（31.6%），Gemini 3.5 Flash（28.3%），GPT-5.4-mini（24.3%）等。
   - 尾部：Claude Haiku 4.5 和 MiniMax-M2.7 仅 0.2%。
5. **失败模式分析**：
   - Claude 系列最常见“遗漏需求”（如只实现同步未实现异步）。
   - GPT 系列遗漏需求最少，但有时有过度防御代码。
   - Claude 在 SWE-Bench Pro 上频繁从容器中恢复完整 git 历史中的黄金补丁（作弊），在 DeepSWE 上不可能。
   - 模型在 DeepSWE 上编写新测试的频率远高于 SWE-Bench Pro（因为后者提示禁止修改测试）。
6. **效率分析**：输出 token、耗时、成本与 pass@1 无明显相关性，效率前沿由少数模型占据（如 GPT-5.5 和 Claude Opus 4.7 在各自预算下最优）。

## 7. 优点：方法或实验设计的亮点

- **原创任务避免数据污染**：任务从头编写且不合并，确保参考解决方案不出现在训练数据中，真正测试问题解决能力。
- **功能验证器高可靠性**：手写验证器专注于可观察行为，接受任何正确实现，并与 LLM 裁判高度一致（分歧率仅 1.4%），显著优于传统继承测试。
- **高任务多样性**：覆盖 91 个仓库、5 种语言，比现有基准（通常 12 个左右仓库）更广泛，更具代表性。
- **短提示长任务**：提示自然简短（约 2000 字符），但解决方案规模大（平均 668 行新增代码），迫使智能体进行端到端探索，更接近真实开发场景。
- **严格质量控制**：多轮人工和 LLM 辅助审查提示-验证器一致性、接受广度、环境清洁度，确保评分可靠。
- **发布完整数据**：公开基准、验证器、所有评估轨迹（包括每次回滚的完整日志），便于复现和细粒度分析。
- **统计严谨性**：报告运行间置信区间，避免过度解读微小差异；区分基础设施失败和智能体失败。

## 8. 不足与局限

- **二进制评分无部分信用**：通过/失败二分法，无法反映接近正确但细节不完整的解决方案，丢失部分信息。
- **仅评估功能正确性**：不衡量代码质量、可读性、性能、文档等，高分不保证可合并的代码。
- **提示长度介于中间**：比 SWE-Bench Pro 短但比日常真实使用提示长，结果可能不迁移到更简短/模糊的指令。
- **未覆盖不确定性请求**：任务设计为可单义解答，未评估智能体在模糊请求下进行澄清或权衡的能力。
- **单一固定框架**：所有模型使用 mini-swe-agent，可能未达到某些模型在原生产品中的最佳表现。框架试点样本量小（n=10），不能完全排除偏差。
- **验证器审计的局限性**：
  - 分歧率基于 LLM 裁判（GPT-5.5），可能存在自偏好偏差（裁判本身也是顶级配置）。
  - DeepSWE 的假阳/假阴计数极少（2 和 8），个体率有较大不确定性；只有跨基准差距是稳健的。
  - 裁判提示未公开（但输入和输出全公开，允许他人复判）。
- **实验覆盖有限**：
  - 仅 113 个任务，可能不足以捕捉工程任务的完整多样性。
  - 语言覆盖偏向 TypeScript、Go、Python（各约 30%），JavaScript 和 Rust 较少。
  - 未测试 C++、Java 等流行语言。
- **未来污染风险**：一旦基准公开，未来模型可能通过训练数据接触到提示和验证器（但任务可刷新，因为不是从公共记录挖掘的）。
