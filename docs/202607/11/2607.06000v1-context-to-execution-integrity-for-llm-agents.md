## 摘要
语言模型代理读取攻击者可写的上下文以完成任务。工具执行需要对受保护接收器字段、接收器解释的有效载荷以及调用事件进行单独的权限检查。上下文到执行完整性（CXI）是针对这种情况的执行边界系统。策略标记受保护的接收器字段，类型化发布将狭窄的验证值从可写上下文传递到特定目的地，不透明数据槽将证据作为数据保存，确定性门仅在字段权限、精确效果授权和调用权限都绑定到同一个动作清单后才允许调用。我们在开放权重字段投影运行、AgentDojo实时情节、代码代理精确效果基准、清单绑定账本故障、提案压力控制以及托管/API兼容性轨迹上评估了CXI。AgentDojo涵盖720个实时情节和1,739个LLM调用；代码代理基准涵盖400个仓库情节，具有精确效果授权和租赁绑定执行，产生231个安全任务完成，零观察到的字段、效果或调用逃逸。会计报告输出解析结果、授权结果和任务质量结果以及准入完整性结果。在评估的接收器中，CXI仅在字段、效果和调用权限绑定到相同动作清单时才允许执行。

## Abstract
Language-model agents read attacker-writable context to solve tasks. Tool execution needs a separate authority check for protected sink fields, sink-interpreted payloads, and the invocation event. Context-to-Execution Integrity (CXI) is an execution-boundary system for this setting. Policies mark protected sink fields, typed releases carry narrow validated values from writable context to specific destinations, opaque data slots keep evidence as data, and a deterministic gate admits a call only after field authority, exact-effect authorization, and invocation authority all bind to the same action manifest.   We evaluate CXI on open-weight field-projection runs, AgentDojo live episodes, a code-agent exact-effect benchmark, manifest-bound ledger faults, proposal-pressure controls, and hosted/API compatibility traces. AgentDojo covers 720 live episodes and 1,739 LLM calls; the code-agent benchmark covers 400 repository episodes with exact-effect authorization and lease-bound execution, yielding 231 safe task completions and zero observed field, effect, or invocation escapes. The accounting reports parser outcomes, authorization outcomes, and task-quality outcomes together with the admission-integrity result. Across the evaluated sinks, CXI admits execution only when field, effect, and invocation authority bind to the same action manifest.

# 论文详细总结

## 1. 核心问题与整体含义（研究动机和背景）

- **问题本质**：LLM 代理（agent）在完成任务时读取攻击者可写的上下文（如 CI 日志、GitHub issues、网页、RAG 块等），这些上下文中的恶意文本不仅能误导模型，更危险的是导致“权限清洗”（authority laundering）：本应仅作为证据的值，却获得了选择工具、授权操作或触发副作用的权限。
- **现有不足**：传统的 schema 验证只检查格式，不检查每个字段的来源和权限；工具调用将整个调用视为可信，忽略了字段级别的权限差异。在 CXI 之前，缺乏一个在副作用发生前执行边界上进行字段级、效果级和调用级权限检查的统一系统。
- **整体含义**：CXI 是一个执行边界系统，通过在工具调用执行前强制检查 **受保护字段**、**接收器解释的效果** 和 **调用事件** 的权限是否绑定到同一个规范动作清单（manifest）上，从而阻止攻击者利用可写上下文控制 LLM 代理的副作用。

## 2. 方法论核心思想与关键技术细节

- **核心思想**：引入“上下文到执行完整性”（CXI），将模型提出的动作在执行边界处分解为三个独立但必须一致的权限链：
  1. **字段权限**：每个保护性接收器字段必须由可信状态或目标作用域的类型化发布（typed release）提供权限。
  2. **效果权限**：接收器解释的有效载荷（如补丁、SQL、Shell 命令）必须通过精确效果授权（exact-effect authorization），即适配器在可信快照下计算的语义效果。
  3. **调用权限**：调用事件本身必须有一个清单绑定的调用能力（invocation capability），且该能力被线性的持久账本消费后产生执行租赁（lease）。
- **关键技术细节**：
  - **类型化发布（D）**：由受信任的解分类器（declassifier）从可写上下文（W）中提取并验证后，生成的作用域仅限单一目的字段的值。
  - **不透明数据槽（O）**：允许复制不可信文本作为证据，但不赋予当前边界的任何权限；后续重新进入副作用边界时视为 W。
  - **确定性门（Gate）**：唯一能够将 schema 有效的候选动作转化为可执行调用的组件。门收到候选动作、字段血统 Π[f]、调用血统 Π_call、策略 P、效果承诺和证据记录后，检查并拒绝或返回执行租赁。
  - **动作清单（Action Manifest）**：绑定动作 nonce、规范动作摘要、适配器 ID/版本、有效主体、可信快照、策略纪元、受保护字段证据摘要、效果授权摘要和调用权限摘要。任何事后突变都会导致摘要变化，从而阻止执行。
  - **精确效果适配器**：对于补丁、文件体、SQL 等，适配器计算接收器在给定快照下将应用的确切效果（如仓库状态增量），CXI 确保执行的效果与授权的效果一致。
- **算法流程**（文字描述）：
  1. 候选动作规范化并生成动作清单。
  2. 对每个受保护字段，检查其血统是否只包含可信 T 或作用域正确的 D，且至少一个原子授权规范值。
  3. 对每个接收器解释的有效载荷，检查是否有精确效果授权。
  4. 解析并消费一个清单绑定的调用能力，获得执行租赁。
  5. 只有三者全部通过，且清单一致，才允许执行副作用。

## 6. 主要结论与发现

- **核心结论**：CXI 在受保护接收器上实现了 0 字段、0 效果、0 调用逃逸。在 AgentDojo 中，所有 204 个授权执行均无逃逸；代码代理基准中 231 个安全任务完成，0 逃逸。

- **机制不可互换**：不同失效路径需要专门边界机制（如总结清洗需要传递性血统，字段复用需要目的作用域，重试控制需要调用能力消费）。
- **实例启示**：CXI 不使文本“一般可信”，而是检查每个具体值、效果和调用是否有针对该清单的权限。

## 7. 优点

- **系统性完整**：CXI 首次将字段、效果、调用三个权限维度统一到同一动作清单下进行门控，覆盖了之前系统（如 PACT、ARGUS、AIRGuard）未同时处理的所有模糊点。
- **粒度精细**：超越整体调用信任，支持字段级血统和精确效果绑定，使得可信上下文仍可携带窄范围值（如文件路径）而不泄露其他权限。
- **兼容性考量**：设计了开放权重路径和托管/API 路径两种证据模型，分别利用可观察的后端证据和主机级构造证据，适应实际部署差异。
- **实验边界清晰**：将提案压力、准入完整性、任务质量分别报告，不混淆不同度量。
- **设计可审计**：策略、解分类器、适配器、门控决策、账本痕迹均为确定性对象，支持复制和审计。

## 8. 不足与局限

- **策略工作成本**：每个接收器都需要手动分类字段、编写解分类器和精确效果验证器，未自动推断策略；策略错误直接导致防护失效。
- **实验覆盖有限**：虽然覆盖多个场景，但策略、快照和任务库固定，外部通用性未验证；例如未包含复杂联网 API、多步代理协作等。
- **验证器完备性**：CXI 强制执行授权，但不保证验证器本身正确或完整。不安全的验证器会导致授权错误。
- **内部透明性**：托管 API 路径无法观察提供商内部的注意力掩码或 KV 缓存，无法提供与开放权重相同的非干扰性保证；仅能声明主机级证据。
- **非功能属性**：未支持精确一次外部交付（需要适配器层面去重），不解决语义任务质量（任务失败非逃逸）。
- **计算成本**：虽然门控开销小（约 0.16 ms 每例），但开放权重的字段投影需要额外推理步骤，实际吞吐量可能下降；托管 API 因往返延迟较大（9-15 秒）。
