## 摘要
编码代理越来越多地为实际软件问题生成拉取请求（PR），但一次性PR生成仍然是开环的：PR在没有系统性审查、诊断或修订的情况下被提出。我们引入了\textbf{SWE-Review}，一个通过代理式代码审查来闭环这一过程的框架。给定一个问题和AI生成的PR，审查代理探索仓库，决定PR是否应被接受，并提供结构化的修订反馈。我们使用提出的\textbf{SWE-Review-Bench}评估这一设置，以衡量审查的正确性和下游修订的有用性。我们进一步整理了\textbf{SWE-Review-Traj}数据集，以研究代理式审查的更广泛应用，并填补开放审查者训练的数据稀缺缺口。实验表明，代理式审查通过生成-审查-修订循环持续改进PR，在修订后的决策准确性和解决率上均优于单轮固定上下文的审查，其能力超越审查本身，可改进问题解决模型，并实现有效且高效的测试时扩展。这些结果将代理式代码审查定位为一种实用机制，推动AI编码代理从一次性PR生成迈向闭环问题解决。

## Abstract
Coding agents increasingly generate pull requests (PRs) for real-world software issues, yet one-shot PR generation remains open-loop: the PR is proposed without systematic review, diagnosis, or revision. We introduce \textbf{SWE-Review}, a framework for closing this loop with agentic code review. Given an issue and an AI-generated PR, a reviewer agent explores the repository, decides whether the PR should be accepted, and provides structured feedback for revision. We evaluate this setting with our proposed \textbf{SWE-Review-Bench} to measure both review correctness and downstream revision usefulness. We further curate \textbf{SWE-Review-Traj} dataset to study broader applications of agentic review and fill the data-scarcity gap for open reviewer training. Experiments show that agentic review continuously improves PRs through a generate-review-revise loop, outperforms single-turn fixed-context review in both decision accuracy and resolve rate after revision, transfers beyond review to improve issue-resolution models, and enables effective and efficient test-time scaling. These results position agentic code review as a practical mechanism for moving AI coding agents from one-shot PR generation toward closed-loop issue resolution.

- **核心问题**：当前AI编码代理为实际软件问题生成拉取请求（PR）时，往往是一次性生成（one-shot），缺乏系统性的审查、诊断和迭代修正，形成“开环”。代码审查本应是闭环的自然机制，但现有自动化审查能力在决策准确性、诊断有用性和端到端评估方面远落后于问题解决模型。
- **研究动机**：随着AI生成的PR数量激增，需要一个可靠的自动审查机制来判断补丁是否正确解决问题，并在不正确时提供可操作的反馈，从而推动从“开环”生成向“闭环”问题解决演进。
- **整体含义**：代理式代码审查不仅是后处理的质量门控，更是实现闭环问题解决的核心机制：可以筛选无效补丁、指导修订，甚至通过训练数据提升问题解决模型本身。

## 2. 方法论

### 核心思想
- **代理式代码审查**：给定仓库、问题描述和AI生成的候选PR，审查代理可以在仓库环境中自由浏览、搜索代码、运行命令，然后输出一个二元决策（接受/请求修改）和结构化的诊断报告（包括缺陷位置、描述和修复建议）。
- **闭环流程**：生成 → 审查 → 修订 → 再审查，直到审查者接受或达到预算上限。

### 关键技术细节
1. **任务形式化**：
   - 输入：仓库代码（固定commit）、自然语言问题、候选PR（diff + title + body）。
   - 输出：`{"decision": "approve" or "request_changes", "summary": {...}, "defects": [...]}`。
   - 审查代理不得访问金标准补丁或隐藏测试。

2. **评估指标**：
   - **CR（Completion Rate）**：是否生成可解析的评审报告。
   - **DA（Decision Accuracy）**：决策与真实补丁解决状态的匹配度。
   - **RRR（Resolve Rate after Revision）**：评审后，被拒绝的补丁由原生成器根据反馈进行一次修订后的最终解决率。

3. **基准构建（SWE-Review-Bench）**：
   - 基于SWE-bench Verified的500个问题。
   - 使用三个不同能力的PR生成器（GLM-5、Qwen3-Coder-30B-A3B、Qwen3-30B-A3B）生成候选PR，分别代表高、中、低质量分布，共1384个PR。

4. **训练数据集（SWE-Review-Traj）**：
   - 从SWE-rebench获取约6000个问题，用同样三个生成器生成14156个候选PR。
   - 用GLM-5（含thinking）作为教师模型进行代理式审查，记录轨迹。
   - 仅保留决策正确的轨迹（教师正确批准或拒绝），共8914条。
   - 进一步通过语义验证（两LLM法官评分）和功能验证（修订成功率）确保诊断质量。

5. **蒸馏训练**：
   - 对Qwen3-8B和Qwen3-30B-A3B进行SFT，训练中掩码错误回合及系统提示。
   - 混合训练：同时使用问题解决轨迹和审查轨迹，训练单一模型承担生成和审查双重角色。

### 算法流程（文字说明）
```
循环直到审查通过或达到最大轮次：
  1. 编码代理根据问题生成/修订补丁。
  2. 审查代理在仓库环境中探索（阅读文件、搜索代码、运行测试等）。
  3. 审查代理输出决策和诊断。
  4. 若为“批准”，则结束循环；若为“请求修改”，则将诊断反馈给编码代理进行修订。
```

## 6. 主要结论与发现

1. **代理式审查显著优于单轮审查**：在三个PR生成器上，代理式审查在DA和RRR上全面超越单轮（diff-only和diff+context）基线，且优势在困难问题上更大（图4）。
2. **闭环迭代持续提升解决率**：对Qwen3-30B-A3B，通过生成-审查-修订循环，解决率从27.5%提升至52.6%（+25.1pp）；对更强大的生成器也有显著提升（图1）。
3. **审查轨迹可有效蒸馏**：对8B模型SFT后，CR从~4%提升至71-84%，DA从近随机提升至67-72%，RRR提升达7.6pp（表2）。
4. **混合训练同时提升审查和生成能力**：添加审查轨迹后，相同数据量下直接解决率提升5.6pp，且同一模型可自审查自修订，最终解决率提升10.6pp（表3）。
5. **审查引导迭代修订是最高效的测试时缩放策略**：相比独立重采样或验证器选最佳，在仅使用2.44个样本平均花费下达到38.4%解决率（图5）。
6. **诊断反馈比二值决策更有价值**：功能验证表明，提供完整诊断的修订成功率（21%）远高于仅提供决策（8%）或无反馈（3%）（表1b）。

## 7. 优点

### 方法设计亮点
- **任务定义合理**：将审查从单轮文本生成改造为多步仓库探索任务，更符合实际代码审查场景。
- **评估指标操作性强**：DA和RRR直接与可执行的补丁验证挂钩，避免了需要人工标注的文本相似度指标。
- **数据构建严格**：轨迹筛选采用决策正确性过滤，并通过语义和功能双重验证确保诊断质量。
- **混合训练思路新颖**：证明审查和生成可以互相促进，符合认知科学中的“教学相长”理念。

### 实验设计亮点
- **分层评估**：按难度、生成器能力、审查范式分别分析，揭示了代理式审查的价值随困难程度递增的规律。
- **测试时缩放对比全面**：涵盖了多种选择策略（审查门控、验证器、批评者）并报告了实际样本消耗效率，结果具有说服力。
- **消融实验细致**：例如诊断质量的功能验证区分了“仅决策”和“带诊断”的效果差异。

## 8. 不足与局限

### 实验覆盖局限
- **PR类型单一**：仅覆盖SWE类错误修复，未涉及新功能实现、重构、文档、迁移、架构变更等常见的PR类型。
- **指标仅关注功能正确性**：DA和RRR只衡量补丁是否通过测试，不评估代码风格、可读性、可维护性、安全性、性能或项目约定。
- **模型和脚手架依赖**：所有实验使用OpenHands-SDK和特定的模型家族（Qwen、GLM、Claude），在其他脚手架（如SWE-Agent）或其他模型族上的表现未知。

### 偏差风险
- **生成器分布偏差**：三个生成器均来自同一生态（Qwen和GLM），未使用其他主流模型（如GPT-4、DeepSeek-Coder）验证，可能高估了方法在通用场景下的效果。
- **教师模型偏差**：轨迹由GLM-5生成，可能引入GLM-5特有的审查风格或错误偏见。尽管进行了语义验证，但验证法官（Claude/GPT）本身也可能存在偏向。
- **单轮审查设置**：单轮基线使用Claude Opus 4.6，但并未像代理式审查那样进行充分的超参数调优（如上下文长度、检索策略），可能低估了单轮审查的上限。
- **测试时缩放对比**：验证器使用8B级别的SWE-Lego-Verifier-8B，而审查器也是8B模型，未比较同样大小的验证器与更大审查器的差异。

### 应用限制
- **Token消耗高**：代理式审查的token消耗约为单轮审查的10倍（表13），在高吞吐场景下成本可能不可接受。虽然附录B.3分析了效率，但未提供与单轮的等价对比。
- **不处理非功能性需求**：审查无法评判性能回归、安全漏洞、兼容性问题（除非有测试覆盖）。
- **假定可执行环境**：方法依赖可运行的Docker环境和测试套件，对于缺乏测试或构建困难的项目不适用。
