## 摘要
LLM代理通过工具调用在现实环境中行动，单个误判行为可能导致不可逆的伤害。标准防护措施是一个守卫模型，它将每个提议的行为标记为安全或不安全，但这种二元视图混淆了两个不同的决策：行为本身是否有害，以及给定用户上下文是否合适。它还以行为类别而非单个实例的粒度运行，产生常规中断，削弱自主性，并训练用户忽略最关键的警报。我们将问题重新定义为关于{执行，询问，拒绝}的逐实例三路路由决策，并通过安全哨兵实例化，这是一种轻量级守卫模型，其推理简化为单个解码调用。一个解码时间阈值允许一个固定的检查点在不重新训练的情况下跨不同风险容忍度的部署重新定位。安全哨兵在整体准确性和与安全相关的召回率上优于广泛的开权重和前沿闭源基线，同时控制两个方向的错误率。

## Abstract
LLM agents act on real-world environments through tool calls, and a single misjudged action can cause irreversible harm. The standard safeguard is a guard model that labels each proposed action as safe or unsafe, but this binary view conflates two distinct decisions: whether the action is harmful in itself, and whether it is appropriate given the user's context. It also operates at the granularity of action categories rather than individual instances, producing routine interruptions that erode autonomy and train users to wave through the most consequential alerts. We reframe the problem as a per-instance three-way routing decision over {EXECUTE, ASK, REFUSE} and instantiate it with Safety Sentry, a lightweight guard model whose inference reduces to a single decoding call. A single decoding-time threshold lets one fixed checkpoint be re-positioned across deployments of differing risk tolerance without retraining. Safety Sentry outperforms a broad set of open-weight and frontier closed-source baselines on overall accuracy and safety-related recall, while controlling both directional error rates simultaneously.

## 1. 核心问题与整体含义（研究动机和背景）

- **现有方法的缺陷**：LLM 智能体通过工具调用与环境交互，一个误判动作可能导致不可逆伤害。当前的标准防护是二元守卫模型，将每个提议的动作标记为“安全”或“不安全”。但这种二元视图混淆了两个不同的决策：动作本身是否有害，以及给定用户上下文是否合适。同时，二元守卫基于动作类别而非单个实例的粒度进行判断，导致频繁、常规的中断，侵蚀了智能体的自主性，并训练用户忽略最关键的警报（报警疲劳）。
- **问题的本质**：许多代理动作既非严格安全也非本质不安全，其适宜性往往取决于用户意图和具体运行时环境。正确的响应不是直接拒绝，而是在执行前请求用户确认。二元安全/不安全框架缺乏中间决策边界。
- **重新定义问题**：作者将代理安全重新定义为每实例的三路路由决策：{执行 (EXECUTE), 询问 (ASK), 拒绝 (REFUSE)}。目标是平衡自主性、安全性和人工监督。

## 2. 方法论

### 核心思想
- **三路路由**：对每个提议的动作 `a_t`，守卫模型输出一个决策 `ŷ ∈ {EXECUTE, ASK, REFUSE}`。守卫的输入是元组 `x = (m, u, d, h<t, a_t)`，其中 `m` 是用户记忆项，`u` 是用户任务，`d` 是应用域，`h<t` 是历史记录。

### 关键技术细节
1. **数据构建**：
   - **真实服务环境**：部署 9 个自托管企业级系统（Gitea, Rocket.Chat, ownCloud, NocoDB, Zammad, ERPNext, OpenEMR, Vaultwarden, Mailu），每个动作标注反映在真实系统上的后果。
   - **任务与用户上下文**：任务来自内部 LLM 生成的指令（覆盖 9 个服务的操作场景）和 5 个外部基准（When2Call, AT-Bench, AgentHarm, TS-Bench, R-Judge）。内部任务配对一个可选的角色记忆，分为谨慎型、许可型、对抗型。
   - **轨迹收集与标注**：首先让 LLM 代理在工具集上完成任务（无安全判断），得到真实工具调用记录。然后对每个动作 `a_t` 和记忆变体 `m`，标注者根据 14 种触发模式（分为三类决策）分配标签。标注采用双 LLM 标注 + 作者仲裁，测试集由作者审计，达到 92% 人类一致性。
2. **模型训练**：
   - 基础模型：Qwen3.5-4B。
   - 扩展分词器：新增三个特殊 token `<|direct_execute|>`、`<|ask_human|>`、`<|refuse|>`。
   - 训练方法：使用 LoRA（rank=16, alpha=32, dropout=0.05）微调所有线性投影，同时解冻并训练新 token 的输入嵌入行，其余嵌入行冻结。损失函数为完成令牌（<think> 块 + 决策 token + JSON 负载）上的交叉熵。训练 3 个 epoch，最大序列长度 6144，bfloat16 精度，学习率 2×10⁻⁴，余弦学习率调度，150 步预热。
3. **阈值路由机制**：
   - 在推理时，对非 REFUSE 子空间，计算重归一化概率 `q_EXECUTE = p_EXECUTE / (p_EXECUTE + p_ASK)`。
   - 给定阈值 `τ ∈ [0,1]`，如果 `q_EXECUTE ≥ τ` 则预测 EXECUTE，否则 ASK。REFUSE 预测保持不变（安全底线）。
   - 一个固定检查点可通过调整 `τ` 适应不同风险容忍度的部署，无需重训。

## 3. 实验设计

### 数据集与场景
- **内部构造**：从 9 个自托管企业服务（Vaultwarden, Zammad, ERPNext, NocoDB, Gitea, OpenEMR, ownCloud, Rocket.Chat）收集 4,575 条轨迹记录，加上 5 个公开基准（When2Call, AT-Bench, AgentHarm, TS-Bench, R-Judge）共 9,203 条步级记录。训练/测试拆分为 7,767/1,436（约 8:2），Mailu 服务完全留作 OOD 评估。

### 基准方法
- **开源小模型**：Llama-3.2-3B-Instruct, Ministral-3B-2512, Gemma-3-4B-IT。
- **Qwen3.5 系列 (尺度控制)**：0.8B, 2B, 4B (pre-SFT), 9B, 27B。

### 评估指标
- **整体质量**：准确率 (Acc)、宏平均 F1 (Macro-F1)、每类 F1。
