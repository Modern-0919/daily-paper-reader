## 摘要
使用工具的LLM智能体大多在假设所有工具正常工作的条件下进行评估。当工具超时、返回过时值或在部署过程中其描述被篡改时，开发人员需要一种可控的方式来复现故障、测试修复并确认修复有效。我们提出AgentCheck，一个开源Web工作台，它将MCP服务器转变为干预面。AgentCheck让智能体在其真实工具上运行，并记录每次工具响应，然后通过故障（12种类型）注入扰动响应后重新运行智能体。匹配的工具调用从缓存中重放，后续工具调用在智能体行为发散后实时运行。这形成了复现-干预-确认循环：开发人员切换缓解措施，针对相同故障重新运行，并观察故障是否消失。评分包含两部分：确定性通过/失败规则，以及用于解释性标签的LLM评判器（基于人工标注验证）。在五个智能体中，最佳通过105/120个场景，最弱仅通过77个。故障通常是无声的、自信地使用错误的工具输出，而非崩溃。在最弱智能体上，重试缓解措施将超时错误故障的成功率从低至30%提高到100%，而过时数据故障无论采取何种缓解措施，成功率仍保持在10次中约3-4次。AgentCheck使这些故障模式在部署前变得可复现、可比较和可验证。

## Abstract
Tool-using LLM agents are mostly evaluated assuming all tools work. When a tool times out, returns a week-stale value, or has its description poisoned in deployment, the developer needs a controlled way to reproduce the failure, test a fix, and confirm the fix worked before deployment. We present AgentCheck, an open-source web workbench that turns an MCP server into an intervention surface. AgentCheck runs an agent against its real tools and records every tool response, then re-runs the agent with the response perturbed by a fault (12 types) injector. Matching tool calls are replayed from cache, and later tool calls go live after the agent diverges. This yields a reproduce-intervene-confirm loop: the developer toggles a mitigation, re-runs against the identical fault, and sees if the failure goes away. Scoring has two parts: deterministic pass/fail rules, plus an LLM judge for interpretive labels, validated against human annotations. Across five agents, the best passes 105/120 scenarios and the weakest only 77. The failures are usually silent, confident use of incorrect tool outputs rather than crashes. On the weakest agent, a retry mitigation raises success on timeout error faults from as few as 30% of cases to 100%, whereas stale-data faults remain near 3-4 of 10 regardless of the mitigation. AgentCheck makes these failure modes reproducible, comparable, and verifiable before deployment.

## 1. 论文的核心问题与整体含义（研究动机和背景）

- **核心问题**：现有的LLM工具使用型智能体（agent）评估基准（如GAIA、SWE-bench、ToolBench）均假设工具正常工作，但在真实部署中，工具可能超时、返回错误、返回过时数据或描述被投毒（poisoning）。这些故障难以复现、诊断和验证修复，导致开发人员无法在部署前系统性地测试智能体在非理想条件下的行为。
- **研究背景**：近期研究（如MCPTox、ToolMisuseBench、PlanBench-XL）表明工具故障对智能体影响显著，例如72.8%的智能体会执行投毒指令。现有工具要么是离线基准（无法测试开发者自己的智能体），要么是监控/日志系统（被动观察，无法注入故障并验证修复）。缺少一个允许开发者主动注入故障、比较轨迹、应用缓解并确认修复效果的闭环工作台。
- **整体含义**：AgentCheck填补了这一空白，通过将MCP服务器作为干预面，实现故障的可控复现、缓解比较和修复验证，使部署前鲁棒性测试成为可能，尤其揭示了静默的**数据质量故障**（而非崩溃）是智能体的主要弱点。

## 2. 论文提出的方法论：核心思想、关键技术细节

- **核心思想**：基于“受控比较”原则：先记录一次智能体在真实工具上的干净运行（clean run），缓存所有工具响应；然后重新运行智能体，但通过故障注入器精确篡改其中一次工具响应（p, g），其余响应从缓存重放；得到干净轨迹和故障轨迹的对比。若有缓解措施，再次以相同故障运行智能体，验证是否修复失败。
- **关键技术细节**：
  - **故障注入引擎**：位于智能体与工具之间，支持12种故障类型，分为三类：
    - A类（工具执行故障）：超时（A1）、API错误（A2）、权限拒绝（A3）、模式漂移（A4）。
    - B类（数据质量故障）：过时数据（B1）、矛盾数据（B2）、错误答案（B3）、空结果（B4）。
    - C类（安全故障）：提示注入（C1）、描述投毒（C2）、虚假声明（C3）、数据泄露（C4）。
  - **缓存重放**：在故障运行中，匹配相同工具和参数的工具调用从缓存返回，只有指定调用点(p)的响应被故障函数g替换。后续调用在智能体行为发散后恢复实时执行。
  - **评分机制**：两部分：
    - 确定性通过/失败检查（primary checks）：基于规则（如是否传播错误值、是否调用未经允许的域名）。
    - LLM评判器（diagnostic labels）：用于输出解释性标签（故障检测、恢复动作、不确定性交流），并经过人类标注验证（Cohen's κ在0.69-0.87之间）。
- **算法流程**（文字描述）：
  1. 干净运行：智能体对任务q使用工具集T，缓存所有响应到字典K。
  2. 故障运行：定义注入点p（工具t, 调用索引i）和故障函数g，重放函数REPLAY在p处用g(r)替换，其余返回K中缓存值。
  3. 执行故障运行得到轨迹τ_fault，计算所有失败的确定性检查集合F。
  4. 若提供缓解措施m，则运行缓解后的智能体m(A)在相同故障(p,g)下，得到τ_mit，检查所有F是否全部通过，给出fix_confirmed布尔结果。

## 6. 论文的主要结论与发现

1. **智能体鲁棒性差异巨大**：最佳配置（DeepSeek）通过105/120，最弱（Llama）仅77/120，表明同一任务在不同模型/执行框架下失效模式差异显著。
2. **静默数据质量故障是最主要弱点**：所有智能体在类别B（过时、矛盾、错误、空）上评分最低（DeepSeek 29/40，Llama 13/40）。这些故障没有错误代码，容易通过乐观基准。
3. **重试缓解对超时有效，但对陈旧数据无效**：在Llama上，重试将超时类故障（A1/A2/A3）的成功率从30%-50%提升至100%（A1）或更高，但B1（过时数据）几乎无改善（4/10→4/10-5/10）。
4. **安全故障（类别C）缓解效果因类型而异**：提示注入（C1）和描述投毒（C2）可通过注入过滤器检测，但数据泄露（C4）需要专门的外泄防御（与地址黑名单无关）。
5. **确定性检查与LLM裁判可互补**：确定性检查提供可重复的通过/失败，LLM裁判提供诊断洞察，但需注意裁判可能过度判断“传播”，因此论文将传播率视为上界。

## 7. 优点：方法或实验设计上的亮点

- **受控比较设计**：通过缓存重放确保干净轨迹和故障轨迹的唯一差异是指定故障，消除了环境噪声，使结果可归因。
- **闭环修复验证**：支持先复现故障，再应用缓解，然后自动重跑确认修复，实现“复现-干预-确认”循环，这是其他工具（如AgentDiagnose、LangSmith）不具备的。
- **高效的故障注入**：12种故障类型覆盖执行、数据质量和安全三类，且提供预配置的120个场景，降低开发者使用门槛。
- **双通道评分**：确定性规则保证结果可重复，LLM裁判提供可解释性，并经过人类标注验证，平衡了精确度与可理解性。
- **开放性**：工作台、场景库、评分器及所有轨迹均以MIT许可公开，可复现和扩展。
- **可视化仪表盘**：并排显示干净、故障、缓解三条轨迹，标记发散点，便于交互式调试。

## 8. 不足与局限

- **单故障注入**：只测试一次注入一个故障，未考虑复合故障或级联故障（如一个工具失败导致后续依赖链断裂）。实际部署中故障常级联。
- **场景覆盖有限**：12种故障类型并非全部可能的故障；B类场景为新编写，可能缺乏真实世界多样性；未覆盖长时域或策略特定故障（如智能体因延迟累积而超时等）。
- **缓解实验通用性受限**：只在Llama上测试缓解措施，未在其他智能体上验证结论的普遍性；且只测试了四种简单缓解（重试、模式检查、注入扫描、全组合），未探索更复杂的缓解（如临时推理、交叉调用一致性）。
- **LLM裁判可靠性问题**：论文明确承认裁判可能过度判断“传播”，将其视为上界（上限估计），且与人类标注者之间存在中等一致性（κ 0.69-0.87），因此需谨慎使用裁判标签作为主要指标。
- **算力与环境透明性不足**：未报告实验所用算力，仅提及使用商业API，可能影响能耗和成本的可复现性。
- **伦理风险**：工具可用于测试安全故障（如数据泄露），可能被滥用；论文声明仅在用户提供的MCP服务器上运行，但缓解措施可能带来虚假安全感（一个通过的修复不能保证通用鲁棒性）。
