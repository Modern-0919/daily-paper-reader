## 摘要
使用工具的LLM智能体主要在假设所有工具都能正常工作的前提下进行评估。当工具超时、返回一周前的陈旧值、或在部署中描述被篡改时，开发人员需要一种可控的方式来复现故障、测试修复方案，并在部署前确认修复有效。我们提出了AgentCheck，这是一个开源Web工作台，它将MCP服务器转变为一个干预接口。AgentCheck让智能体在其真实工具上运行，并记录每个工具的响应，然后通过故障（12种类型）注入器扰动响应后重新运行智能体。匹配的工具调用从缓存中重放，而后续的工具调用在智能体偏离后实时进行。这形成了一个复现-干预-确认循环：开发人员切换缓解措施，针对相同的故障重新运行，并观察故障是否消失。评分包含两部分：确定性通过/失败规则，以及一个用于解释性标签的LLM评判器，该评判器根据人工标注进行了验证。在五个智能体中，表现最好的通过了105/120个场景，最弱的仅通过了77个。故障通常是静默的，即自信地使用不正确的工具输出而非崩溃。在最弱的智能体上，重试缓解措施将超时错误故障的成功率从低至30%提升至100%，而陈旧数据故障无论采取何种缓解措施，成功率仍保持在每10个约3-4个。AgentCheck使得这些故障模式在部署前可复现、可比较、可验证。

## Abstract
Tool-using LLM agents are mostly evaluated assuming all tools work. When a tool times out, returns a week-stale value, or has its description poisoned in deployment, the developer needs a controlled way to reproduce the failure, test a fix, and confirm the fix worked before deployment. We present AgentCheck, an open-source web workbench that turns an MCP server into an intervention surface. AgentCheck runs an agent against its real tools and records every tool response, then re-runs the agent with the response perturbed by a fault (12 types) injector. Matching tool calls are replayed from cache, and later tool calls go live after the agent diverges. This yields a reproduce-intervene-confirm loop: the developer toggles a mitigation, re-runs against the identical fault, and sees if the failure goes away. Scoring has two parts: deterministic pass/fail rules, plus an LLM judge for interpretive labels, validated against human annotations. Across five agents, the best passes 105/120 scenarios and the weakest only 77. The failures are usually silent, confident use of incorrect tool outputs rather than crashes. On the weakest agent, a retry mitigation raises success on timeout error faults from as few as 30% of cases to 100%, whereas stale-data faults remain near 3-4 of 10 regardless of the mitigation. AgentCheck makes these failure modes reproducible, comparable, and verifiable before deployment.

## 1. 核心问题与整体含义（研究动机与背景）
- **现有评估的盲区**：主流 LLM 代理基准（如 GAIA、SWE-bench、ToolBench）均假设所有工具正常工作。实际部署中，工具可能超时、返回陈旧数据、描述被投毒等，这些故障很少被系统性地测试。
- **开发者痛点**：当代理在生产中因工具故障而失败时，开发者需要一种**可控、可复现、可验证**的方式：复现故障 → 应用修复 → 确认修复生效。现有工具要么只能离线评测（如 ToolMisuseBench），要么只能事后观察轨迹（如 LangSmith），缺乏自动化的故障注入与修复确认闭环。
- **论文意义**：AgentCheck 将 MCP 服务器转化为“干预界面”，让开发者在浏览器中完成“复现-干预-确认”循环，使故障模式在部署前变得可复现、可比对、可验证。

## 2. 方法论：核心思想与关键技术
### 2.1 核心思想
- **可控比较（Controlled Comparison）**：首先记录一次干净的代理运行轨迹（所有工具响应真实），然后在同一任务、同一工具缓存下，仅修改一个工具响应（注入故障），重新运行。通过对比干净运行与故障运行，故障所导致的行为差异被唯一归因。
- **复现-干预-确认循环**：开发者先复现故障，然后应用缓解措施（如重试、过滤等），再对同一故障重新运行，观察失败是否消除。

### 2.2 关键技术细节
- **故障注入引擎**：位于代理与工具之间，支持 12 种故障类型，分为三类：
  - **A 类 – 工具执行故障**：超时、API 错误、权限拒绝、模式漂移。
  - **B 类 – 数据质量故障**：陈旧数据、矛盾数据、错误答案、静默空结果。
  - **C 类 – 安全故障**：提示注入、描述投毒、虚假声明、数据外泄。
- **缓存与重放机制**：干净运行时，所有工具响应存入缓存；故障运行时，匹配的工具调用从缓存中重放，仅选定的注入点被修改；后续调用（代理偏离后）实时执行真实工具。
- **算法流程（文字说明）**：
  1. 调用代理执行任务，记录所有工具响应 → 得到缓存 K。
  2. 定义故障点（工具、参数索引）和注入器。
  3. 以重放函数运行代理：对于每个工具调用，若匹配注入点则返回注入后的响应，否则从缓存返回。
  4. 运行后，根据故障类型执行确定性通过/失败检查。
  5. （可选）对代理包裹缓解措施，重新执行第 3 步，确认修复是否成功。
- **交互式仪表盘**：在浏览器中展示干净/故障/缓解三轨并排对比，标注首次发散点，支持选择模型、注入场景、查看诊断标签。
- **评分机制**：
  - **主要检查（Primary Pass/Fail）**：基于故障类型的确定性规则（如“超时后不得编造结果”）。
  - **诊断标签（Diagnostic Label）**：LLM 评判器（Claude 4.5 Haiku）对轨迹进行四类标签（故障检测、恢复动作、不确定性沟通、传播）。LLM 评判器经人工标注验证（Cohen’s κ 0.69–0.87）。

## 6. 主要结论与发现
- **得分差异大**：最好 DeepSeek 105/120，最弱 Llama 77/120。这种差距在传统全好工具 benchmark 中会被掩盖。
- **静默数据质量故障是最大弱点**：所有代理在 B 类（数据质量）上表现最差（DeepSeek 29/40，Llama 13/40）。这些故障不触发错误码，容易导致代理自信地使用错误数据。
- **缓解效果因故障类型而异**：
  - **重试缓解**：对工具执行故障（A1/A2/A3）提升巨大（如 A1 从 30%→100%，A2 从 50%→100%）。
  - **陈旧数据（B1）、矛盾数据（B2）、错误答案（B3）** 几乎不受缓解影响（仍约 3–4/10）。
  - **数据外泄（C4）** 未被缓解工具覆盖，改善为 0。
  - **模式检测（Scan）** 对模式漂移（A4）、静默空结果（B4）有一定帮助。
- **故障机制**：失败多为“自信使用错误输出”而非崩溃，属于静默故障。

## 7. 优点
- **可控复现性**：通过缓存与重放机制，故障效果可完全归因，排除了环境噪声。
- **修复验证闭环**：支持对同一故障重新运行缓解后的代理，输出 `fix_confirmed` 记号，这是现有工具缺失的功能。
- **系统化故障库**：12 种故障类型覆盖执行、数据、安全三类，可扩展。
- **公平比较**：所有代理共享相同的固定工具响应，使得差异仅来自模型/框架。
- **开源与交互界面**：浏览器使用，降低门槛；附带视频演示和代码仓库。
- **评分可靠性**：确定性检查为主，LLM 评判器为辅助，且有人工验证。

## 8. 不足与局限
- **单故障注入**：未评估级联故障或多故障同时发生，与现实可能不完全匹配。
- **故障类型不全面**：12 种故障未覆盖所有政策特定失败模式（如拒绝服务、错误率突然升高、非连续时序等）。
- **LLM 评判器可靠性上限**：即使经过验证，LLM 评判器仍可能产生误判（尤其传播率被作者声明为“上界”）。
- **场景库部分手工**：B 类场景为新创作，存在人为偏差风险；未与真实生产日志对比验证故障的真实性。
- **缓解实验范围有限**：仅在 Llama 上进行完整消融，其他代理的缓解效果未知。
- **适用性限制**：要求代理与工具通过 MCP 连接；不适用于非 MCP 架构或自定协议。
- **伦理风险**：安全故障注入（如数据外泄）可能被滥用，但作者强调仅开发者在自有 MCP 服务器上运行。
