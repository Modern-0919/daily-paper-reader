## 摘要
我们提出了AgentLens，一个用于交互式代码代理的生产评估基准。大多数代码代理基准将一次运行简化为单个比特——任务是否通过？——但实际使用这些代理的人们体验的是整个轨迹：代理如何遵循指令、使用工具、验证自身工作、从错误中恢复以及在此过程中与用户交互。AgentLens评估了整个轨迹。它将形式化验证（存在客观检查）与LLM编写的轨迹回顾及并排比较相结合，使得每次运行都能产生关于评分原因的可读解释。这使得AgentLens不仅仅用于模型排名：我们用它来诊断模型行为、比较我们自身代理的连续版本，并在夜间评估流水线中捕获产品回归。我们将该基准以开源形式发布在https://github.com/agent-lens/agent-lens-bench。

## Abstract
We present AgentLens, a production-assessed benchmark for interactive code agents. Most code-agent benchmarks reduce a run to a single bit -- did the task pass? -- but the people who actually use these agents experience the entire trajectory: how the agent follows instructions, uses its tools, verifies its own work, recovers from mistakes, and talks to them along the way. AgentLens evaluates that whole trajectory. It pairs formal verification, where an objective check exists, with LLM-written trajectory reviews and side-by-side comparisons, so that each run yields a readable explanation of why the score is what it is. This makes AgentLens useful for more than ranking models: we use it to diagnose model behavior, compare successive versions of our own agent, and catch product regressions in a nightly evaluation pipeline. We release the benchmark as open source at https://github.com/agent-lens/agent-lens-bench.

- **现有问题**：大多数编码代理基准（如SWE-bench）仅用一个二进制通过/失败位来评估任务完成情况，这无法捕捉用户在实际使用中体验到的完整交互轨迹，包括指令遵循、工具使用、自我验证、错误恢复以及与用户的沟通质量等。
- **研究动机**：对于日常使用的编码助手，二进制指标无法反映可靠性、过度自信、不愉快的交互或与用户意图的细微偏差；而即使任务最终失败，过程中也可能有大量有用的行为。此外，许多任务（如编写文档）没有自然的二进制结果。因此需要更全面的评估方法。
- **核心含义**：论文提出了AgentLens——一个生产评估的交互式编码代理基准，评估完整的交互轨迹而非仅最终状态，并结合形式化验证与LLM撰写的轨迹回顾，提供可读的评分解释，用于模型诊断、版本比较和回归检测。

## 2. 论文提出的方法论

- **核心思想**：评估对象是完整的交互轨迹（包括用户消息、代理回复、工具调用、文件编辑、命令执行、验证尝试、最终答复和仓库最终状态），使用LLM评审评分，辅以形式化验证，提供多维度的诊断信息。
- **关键技术细节**：
  - **轨迹回顾协议**：每个轨迹由LLM评审员（judge）在五个维度上评分：最终结果（END RESULT）、指令遵循（INSTRUCTION COMPLIANCE）、陷阱/可修复失败（PITFALLS）、愉悦度（PLEASANTNESS）、工具调用（TOOL CALLS）。评分采用小序数量表（0/0.5/1），并提供带有证据引用的书面回顾。
  - **形式化验证**：包括仓库状态检查、正则表达式检查、测试执行、命令输出检查、静态分析等，提供客观证据。
  - **质量指数（QI）**：五个评审维度得分和形式化验证得分的未加权平均值。
  - **并排比较（Side-by-Side）**：用于比较两个代理在相同任务上的表现，基于预计算的单轨迹回顾和原始轨迹，输出比较性解释和结构化证据行。
  - **用户模拟器**：使用LLM模拟三种人格的用户（默认、有帮助、有毒），以测试代理在不同交互风格下的鲁棒性。
- **公式/算法流程**：质量指数公式 $Q(a) = \frac{1}{|M|} \sum_{m \in M} s_m(a)$，其中 $s_m(a)$ 是代理 $a$ 在指标 $m$ 上的聚合得分（0-100）。并排比较的评分基于陷阱负担总分的差值计算，映射到[-5,5]的整数分数。

## 6. 论文的主要结论与发现

- **主要结论**：AgentLens能够区分代理在多个维度上的表现，不仅仅是最终任务成功；它揭示了代理的不同行为模式（例如，一些代理更遵循指令但最终结果一般，另一些则在工具使用和验证方面更可靠）。并排比较和评审分析能有效诊断回归和模型差异。
- **关键发现**：
  - 质量指数（QI）与公共基准中的代理基准（如APEX-Agents）高度相关（ρ=0.82），但与推理和指令遵循基准（如IFBench）呈负相关，表明AgentLens测评的是不同的能力维度。
  - 某些模型（如Gemini 3.1 Pro Preview）虽被宣传为前沿，但在实际代码代理任务中表现不佳，原因是代理循环可靠性问题而非代码生成能力弱。
  - DeepSeek V4 Pro和Flash性能相近但策略不同：Pro更注重控制和安全性，Flash更注重完成度。
  - 评审能发现二进制指标掩盖的问题，例如Kimi K2.6的低分主要是提供商侧的JSON解析错误而非能力不足；并行工具调用中的竞态条件通过陷阱指标检测出来。

## 7. 优点

- **全面的轨迹评估**：评估完整交互轨迹而非仅最终结果，覆盖五个质量维度，提供诊断信息而非单一分数。
- **可读的评审**：每个分数附带书面评审和证据引用，便于开发者快速定位问题（例如图1示例）。
- **并排比较**：支持模型对比、特征评估和回归检测，特别适合产品开发中的夜间回归测试。
- **稳健性检查**：进行了顺序偏差、评审员自偏好、重复运行方差等分析，提升可信度。
- **可扩展性**：基准设计为可扩展（添加新任务、新指标、新代理接口），支持不同IDE和CLI代理。
- **实际生产场景**：任务来源于真实开发者工作流和匿名使用数据，具有实际相关性。
- **开源发布**：代码、基准、报告示例公开，促进复现和社区使用。

## 8. 不足与局限

- **语言限制**：当前仅支持Java，虽然论文声称可适应其他语言，但未提供实际证据。
- **第三方API依赖**：多个模型通过OpenRouter等第三方API访问，延迟、路由、模型版本不受控制，可能影响结果公平性。
- **成本高昂**：单次运行成本可超100美元，限制大规模实验的可行性和复现。
- **评审有效性**：LLM-as-judge的可靠性依赖于内部实验（未发表正式的一致性研究），且评审员存在自偏好（尤其在愉悦度维度），可能影响排名，尤其在顶级模型比较时。
- **基准规模较小**：仅16个任务32条轨迹，虽然论文声称经验上低方差，但可能不足以全面代表开发者的所有工作类型。
- **塔斯克设计可能偏向自身系统**：AgentLens最初围绕作者的编码助手构建，某些任务或集成假设可能对其更自然，尽管已开源以缓解此问题。
- **缺乏多语言、多领域覆盖**：仅限于Java生态系统，未涉及其他语言（如Python、JavaScript）或非Java框架。
