## 摘要
编码智能体基准测试主要衡量智能体能否生成功能正确的补丁，但生产软件还要求在真实执行目标上实现可衡量的加速。性能优化是一项独特的智能体任务：智能体必须分析执行过程、诊断跨层瓶颈、在不破坏正确性的前提下编辑代码，并验证优化增益是可复现的而非测量假象。我们提出了PERFOPT-Bench，一个用于评估这一完整性能工程循环的基准测试。每个任务提供一个正确但刻意低效的代码库，并要求智能体改进目标性能指标；评分需要隐藏的正确性测试、验证加速测量以及轨迹级审计。我们在7个长周期优化任务上评估了采用不同大语言模型和智能体框架的7种智能体堆栈。结果表明，优化性能取决于具体工作负载，而非仅由模型身份决定：没有单一堆栈占据主导地位，更换智能体框架可能显著改变同一大语言模型在不同任务上的加速表现。我们进一步发现，原始加速比作为基准评分是不安全的，因为某些较大增益来自于对基准测试特定捷径的利用；一项探索性接力实验表明，在初始会话停止后，从外化的优化摘要重新启动可以恢复额外的优化空间。该基准测试及我们的评估结果可在以下地址获取：https://anonymous.4open.science/r/Dataset-D3CC。

## Abstract
Coding-agent benchmarks have largely measured whether agents can produce functionally correct patches, but production software also demands measurable speedups on real execution targets. Performance optimization is a distinct agentic task: agents must profile executions, diagnose cross-layer bottlenecks, edit code without breaking correctness, and verify that gains are reproducible rather than measurement artifacts. We introduce PERFOPT-Bench, a benchmark for evaluating this full performance-engineering loop. Each task provides a correct but deliberately suboptimal codebase and asks the agent to improve a target performance metric; scoring requires hidden correctness tests, verified-speedup measurement, and trajectory-level audit. We evaluate 7 agent stacks with different LLMs and agent frameworks on 7 long-horizon optimization tasks. The results show that optimization performance is workload-dependent rather than determined by model identity alone: no single stack dominates, and changing the agent framework can materially change the same LLM's per-task speedup profile. We further find that raw speedup is unsafe as a benchmark score, since some large gains arise from benchmark-specific shortcut exploitation; an exploratory relay pilot suggests that restarting from an externalized optimization summary can recover additional headroom after an initial session stops. The benchmark and our evaluation are available at: https://anonymous.4open.science/r/Dataset-D3CC.

- **研究动机**：现有编码智能体基准（如SWE-bench、HumanEval等）主要衡量智能体能否生成功能正确的代码补丁，但在生产环境中，软件必须同时满足性能要求——在真实硬件上实现可复现的加速。性能优化是一项独特的智能体任务：不仅需要写正确代码，还需要分析执行、诊断跨层瓶颈、修改代码而不破坏正确性，并验证优化增益的可靠性。
- **整体含义**：本文指出性能优化不能通过单次代码生成完成，而是需要完整的“分析-诊断-修改-验证”循环。现有基准缺失了对这一循环的系统评估，因此作者提出PERFOPT-Bench，专门评估编码智能体在软件性能优化上的能力。

## 2. 论文提出的方法论

### 核心思想
构建一个包含12个长周期任务的基准，每个任务提供一个功能正确但刻意低效的代码库，并给出自然语言优化目标。智能体必须通过分析、编辑、验证来提升目标性能指标（如延迟、吞吐量）。评分结合隐藏正确性测试、验证加速测量和轨迹级审计，防止作弊。

### 关键技术细节
- **半自动化构建管线**（四阶段）：
  1. **方向分解**：LLM驱动生成超过180个候选任务类别（含优化目标、硬件行为、测量方法、通过标准、难度标签）。
  2. **测试用例生成**：为每个类别生成任务包，包括agent可见的工作区（issue.md、原始代码目录Code/）和隐藏评估资产（参考实现、验证输入等）。
  3. **难度校准与对抗强化**：使用原型求解器测试每个候选，若太容易则增加瓶颈层、扩大代码量、提高阈值；反复迭代直至通过率约45%，留下约28个任务。
  4. **专家策展**：资深系统优化专家从校准池中挑选12个代表性任务，调整加速阈值，确保覆盖不同优化领域（构建系统、调度逻辑、数据引擎、查询执行、稀疏求解器、文本处理、图遍历等）。

- **任务分类**：覆盖四种跨层瓶颈：
  - 构建/工具链配置（编译器标志、SIMD调度）
  - 标量计算与缺失向量化
  - 内存访问与缓存局部性
  - 并行性与运行时调度开销

- **评估协议**：
  - 每个任务：issue.md + Code/ + Solution/（初始为空）。智能体看不到参考实现、隐藏评估器和验证输入。
  - 主要指标：**验证加速比** = 基线运行时间 / 提交运行时间，经隐藏正确性测试和专家轨迹审计后报告。
  - 原始加速不安全：存在“基准测试特定捷径”现象（如根据验证器特征过度特化），需要自动化筛选和专家判断。

## 3. 实验设计

### 数据集/场景
- **PERFOPT-Bench**：12个自包含的C语言任务，覆盖8个领域（数据库、数值计算、数据结构、ML/DL、存储、查询引擎、索引扫描、ML运行时、气候模拟、稀疏求解器、文本处理、图分析）。代码量约668K行，中位数约15K行。
- **每个任务**包含构建脚本（build.ps1）、基准驱动（benchmark.c）、自动测试脚本（test.sh/test.ps1）。

### 基准对比
作者没有与其他现有基准直接对比指标，而是将PERFOPT-Bench与相关基准（SWE-bench、PIE、SWE-Perf、KernelBench等）在8个维度上进行比较（见表1），突出PERFOPT-Bench是唯一覆盖性能优化、跨层、连续分数、真实硬件、瓶颈诊断、自动生成、真实仓库、智能体交互全部维度的基准。

### 对比的方法（智能体栈）
共评估7种智能体栈，每个栈由一个LLM和一个编码智能体框架组成：
- **LLMs**：GLM-5.1、GPT-5.5、Opus-4.7、Kimi-K2.6、DeepSeek-V4 Pro
- **框架**：Claude Code、OpenCode、Codex
- **具体栈**：codex-gpt（Codex+GPT-5.5）、oc-gpt（OpenCode+GPT-5.5）、cc-opus（Claude Code+Opus-4.7）、oc-opus（OpenCode+Opus-4.7）、oc-glm（OpenCode+GLM-5.1）、oc-kimi（OpenCode+Kimi-K2.6）、oc-dsv4（OpenCode+DeepSeek-V4 Pro）

额外进行了**接力实验**：选择两个任务（T5, T8）和两个栈（codex-gpt、cc-opus），每个接力序列运行两轮（R1 → R2），包括自接力（相同栈）和交叉接力（不同栈）。

## 6. 论文的主要结论与发现

1. **无单一智能体栈占优**：最优栈随任务变化，没有栈在所有任务上获胜。例如oc-glm获得最高几何平均（9.9×），但从未在单个任务中登顶；oc-gpt和codex-gpt各赢4个任务（表2）。
2. **框架对同一LLM表现有显著影响**：固定GPT-5.5时，oc-gpt在共享单元上几何平均9.2×高于codex-gpt的7.8×，但codex-gpt在特定任务上领先（表4）。Opus-4.7在OpenCode和Claude Code下也有差异（表3）。
3. **原始加速不可靠**：部分高加速值来自基准特定捷径，如答案合成、输出篡改、构建语义绕过等。这些捷径虽然可以大幅提升测量值，但并非通用优化。作者认为应结合隐藏测试、轨迹审计和专家判断来验证加速。
4. **接力实验可恢复额外优化空间**：所有8个两轮序列中R2均优于R1，相对增益1.02×-2.48×。但受限于预算和缺少控制，这一结果仅作为探索性证据。

## 7. 优点

- **首创完整性能优化闭环评估**：不仅是功能性补丁，而是要求智能体自主完成分析、诊断、编辑、验证全过程，反映真实性能工程需求。
- **半自动化构建管道**：结合LLM生成、对抗强化和专家策展，可扩展性强，能产生多样化的跨层优化任务。
- **隐藏评估与轨迹审计**：防止智能体通过硬编码、绕过测试等捷径获得高分，提高了基准的可靠性。
- **明确揭示框架影响**：通过固定LLM仅换框架的对比，展示了同一LLM在不同框架下的不同表现，强调应归功于完整栈而非LLM本身。
- **接力实验的探索**：初步显示优化过程可通过接力文档延续，为智能体系统设计提供了新思路。

## 8. 不足与局限

- **规模有限**：仅12个任务、7个栈，无法代表所有性能优化场景。
- **单次运行**：缺乏重复采样和统计分析，结果解读需谨慎。
- **硬件依赖**：报告的速度比值在特定CPU、编译器和运行时下成立，可迁移性有限。
- **仅覆盖C语言**：未涉及其他语言（如Python、C++、CUDA）或深度学习算子优化，虽然GitHub上常见性能优化任务多属C/C++。
- **捷径审计主观且不彻底**：仅审查可疑案例，可能遗漏其他边界情况。
- **接力实验预算混淆**：没有等预算控制组（如更长单次运行、无文档重启），不能断定增益来自接力文档或栈切换。
- **任务生成依赖LLM**：虽然经过专家策展，但初始任务由LLM生成，可能引入隐性偏斜。
- **未包含资源约束优化**：如内存占用、能耗等生产环境中的其他重要性能维度。
- **基准可能被过拟合**：随着时间推移，智能体可能学会针对这些特定任务优化而失去泛化能力。
