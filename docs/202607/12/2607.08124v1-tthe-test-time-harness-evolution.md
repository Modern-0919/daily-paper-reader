## 摘要
LLM智能体的行为不仅取决于底层模型，还取决于其运行流程：即可执行程序，用于构建上下文、调用工具、验证中间结果以及从失败中恢复。现有方法在部署前优化此类流程，在训练或开发数据中搜索固定的智能体工作流，然后在测试时冻结。这限制了当测试分布、失败模式或工具交互与开发期间所见不同时的适应性。我们提出是否可以在评估过程中仅使用智能体在测试输入上产生的无标签执行轨迹来优化流程。我们引入测试时流程进化（TTHE），将可执行流程视为测试时适应的状态。在评估过程中，TTHE维护一个候选流程群体，并通过一个智能体提议者根据其执行轨迹进行推理来优化它们，无需黄金标签或任务特定监督；然后一个评判者根据执行导出的代理信号提交改进后的流程，选定的程序持续用于后续输入。关键的是，TTHE不更新模型权重、不需要黄金标签，也不训练单独的适应模型：求解器、提议者和评判者是围绕相同冻结LLM的不同角色和流程，因此所有适应都通过对周围程序的改变来实现。在文本到SQL、竞赛编程、软件工程、数据科学编码和智能体工具使用任务中，TTHE改进了固定的ReAct风格基线流程，产生了持久的、可检查的改进，而不是预先搜索的工作流或每次查询重试。这些结果将LLM智能体的测试时适应重新定义为对可执行控制程序的进化，并将执行导出的代理可靠性确定为鲁棒无监督智能体改进的核心挑战。

## Abstract
The behavior of an LLM agent is determined not only by the underlying model, but also by its harness: the executable program that constructs context, invokes tools, verifies intermediate results, and recovers from failures. Existing approaches optimize such harnesses before deployment, searching training or development data for a fixed agent workflow that is then frozen at test time. This limits adaptation when the test distribution, failure modes, or tool interactions differ from those seen during development. We ask whether the harness can instead be optimized during evaluation itself, using only the unlabeled execution traces the agent produces on the test inputs. We introduce Test-Time Harness Evolution (TTHE), which treats the executable harness as the state of test-time adaptation. During evaluation, TTHE maintains a population of candidate harnesses and refines them through an agentic proposer that reasons over their execution traces, without gold labels or task-specific supervision; a judge then commits an improved harness from execution-derived proxy signals, and the selected program persists to govern subsequent inputs. Crucially, TTHE does not update model weights, require gold labels, or train a separate adaptation model: solver, proposers, and judge are different roles and harnesses around the same frozen LLM, so all adaptation occurs through changes to the surrounding program. Across text-to-SQL, competitive programming, software engineering, data-science coding, and agentic tool-use tasks, TTHE improves fixed ReAct-style baseline harnesses, yielding persistent, inspectable improvements rather than a pre-searched workflow or per-query retries. These results recast test-time adaptation for LLM agents as evolution over executable control programs and identify execution-derived proxy reliability as a central challenge for robust unsupervised agent improvement.

- **问题**：LLM 智能体的行为不仅由冻结的模型权重决定，还受其**可执行流程（harness）** 控制——即构造上下文、调用工具、验证中间结果和从失败中恢复的程序。现有方法在部署前针对开发数据搜索一个固定工作流并冻结，导致测试分布、失败模式或工具交互发生变化时无法适应。
- **动机**：评估过程本身会产生丰富的操作证据（执行轨迹），包括模型调用、工具输出、错误信息等，这些证据无标签、无需黄金参考，却蕴含如何改进流程的线索。但现有方法（如微调、测试时训练、自调试、预搜索工作流）要么更新模型权重，要么只修改单次响应，要么在评估前固定流程，均未能利用评估时的执行轨迹**持久地**改进可执行程序。
- **核心问题**：在模型权重冻结、无外部标签或修复信号、无额外训练模型的情况下，能否仅靠智能体自身的执行轨迹，通过修改其周围的 harness 在评估过程中实现适应？

## 2. 方法论核心思想、关键技术细节、公式/算法流程
### 核心思想
- **测试时 Harness 进化（Test-Time Harness Evolution, TTHE）**：将可执行 harness 视为测试时适应的状态，在评估过程中维护一个候选 harness 群体，通过基于执行轨迹的**提议者（proposer）** 和**评判者（judge）** 迭代优化，选择出的 harness 持续用于后续输入。整个循环不更新模型权重、不使用黄金标签。
- 求解器、提议者和评判者由**同一冻结 LLM** 的不同角色和 harness 实现（骨干模型为 deepseek-v4-flash），适应完全通过改变周围程序而非权重更新或额外训练模型实现。

### 关键技术细节
1. **Harness 定义**：一个可执行的 Python 类，控制模型调用、工具使用、验证和恢复逻辑。搜索空间为所有可执行的智能体程序（不限于提示模板或固定工作流）。
2. **优化信号**：无黄金标签，从执行轨迹中提取三类轻量级代理信号：
   - **执行健康度** s ∈ {0,1}：输出无错误且格式一致。
   - **往返一致性** b ∈ [0,1]：将输出逆转为描述并与输入匹配，检测语义漂移。
   - **公开测试通过率** p ∈ [0,1]：若任务提供公开测试，输出通过的比例。
   - 信号与原始轨迹一同提供给提议者和评判者，而非合并为单一标量。
3. **算法流程（Algorithm 1）**：
   - 初始化 harness H = ReAct 基线。
   - 对每个未标注批次 X（大小 B）：
     - 执行当前 H 获取轨迹和缓存输出。
     - 从 H 复制 G 个分支。
     - 对 R 轮进化：
       - 每个分支的提议者（编码智能体）读取所有分支的轨迹，编辑自己分支的父 harness 生成子 harness（无效子代回退父代）。
       - 执行新生成的子 harness，更新轨迹。
     - 最终获得 G 个分支，评判者根据轨迹和代理信号选择其中一个作为新 H，并提交至下一批次（不重新执行）。
     - 使用黄金标签仅对已选 harness 进行测量（transductive 评分）。
4. **多样性设计**：G 个分支被引导向不同编辑方向（保守修复、探索、对抗），可互相借鉴策略，避免过早收敛。

## 6. 主要结论与发现
- **TTHE 在所有域上一致提升固定基线**：BIRD +38.0pp（12→50%）、LiveCodeBench +8.3pp（30→38.3%）、SWE-bench +15.0pp（20→35%）、DS-1000 +6.0pp（38→44%）、claw-eval +20.9 分（48.9→69.8%）（均为硬子集）。
- **进化后的 harness 产生可检查的策略**：例如 SQL 值接地、日期修复、SWE-bench 的“复现-定位-修复-验证”流程和反瘫痪机制等，可直接读取源代码。
- **搜索预算非单调**：G=3、R=3 最优，单分支（G=1）更多回合反而退化；更多并行分支增加有用策略存活概率。
- **批大小非单调**：B=10 最优，更小或更大批次均降低效果。
- **跨模型有效**：不同骨干（MiMo、Kimi）均提升，与模型无关。
- **关键瓶颈**：选择后悔（池内 Oracle 比实际判决高 14pp）和覆盖不足（生成候选从未覆盖约 15 个问题）；代理信号不完美导致评判者有时提交较差策略。
- **位置**：测试时适应重新定义为可执行程序的进化；代理可靠性是进一步鲁棒性的核心挑战。

## 7. 优点
- **方法创新**：将测试时适应从权重更新转为可观察、可编辑的程序代码，开启 harness 作为适应状态的新视角。
- **无监督适应**：完全无需标签、不更新权重、不引入外部训练模型，仅利用执行轨迹，实用性强。
- **可解释性与可审计性**：进化出的 harness 为可读源代码，可直接分析策略源于哪些失败模式。
- **跨泛化**：在多种执行型任务上均有效，且扩展到工具使用工作流。
- **深入消融与诊断**：不仅报告增益，还细致分析搜索预算、批大小、选择后悔，使研究成果具有指导意义。
- **公平性尝试**：使用 hard 子集避免饱和；不同模型验证非模型特有。
