## 摘要
我们提出AfterVibe，一个从vibe编码会话中恢复自然语言规范的框架。给定一个代码产物及其产生的对话轨迹，AfterVibe使用LLM提取一个抽象的自然语言规范，捕获开发者的意图，并通过再生测试进行验证：第二个盲AI agent仅从该规范重新实现产物，得到的代码通过多层验证流水线与原始代码进行评分。因此，规范质量由agent能否重新生成通过的代码来衡量；如果验证器认为实现等价，则规范被认为是强的，否则会迭代改进。通过对一家公司内部编码会话中的72个真实vibe编码项目评估AfterVibe，我们发现其恢复的规范在设计上是抽象的——捕获行为意图而不规定实现——但依然是强的。多次独立再生实现了5.06的高平均再生分数（满分6.0），同时在其细节上保持多样性，确认了规范约束了什么而不过度规定如何做。除了优于现有手工编写的描述外，规范还可以通过迭代进一步加强到5.74分。一个实际意义是，在AI生成的代码超出常规代码审查速度时，规范——而非代码——可能成为人工审查的主要产物和记录来源。

## Abstract
We present AfterVibe, a framework that recovers natural-language specifications from a vibe coding session. Given a code artifact and the conversation trajectory that produced it, AfterVibe uses an LLM to extract an abstract natural-language specification capturing the developer's intent, and validates it through a regeneration test: a second, blind AI agent re-implements the artifact from the spec alone, and the resulting code is graded against the original through a multi-tier validation pipeline. Spec quality is thus measured by whether an agent can regenerate passing code; if the verifiers deem the implementations equivalent the spec is considered strong, otherwise it is iteratively refined. Evaluating AfterVibe on 72 real-world vibe-coded projects from a company's internal coding sessions, we find that its recovered specs are abstract by design-capturing behavioral intent without dictating implementation-yet strong. Multiple independent regenerations achieve a high mean regeneration score of 5.06 out of 6.0 while remaining diverse in their details, confirming that the spec constrains what without over-prescribing how. Besides outperforming existing human-authored descriptions, the specs can be further strengthened iteratively to a score of 5.74. A practical implication is that specifications-not code-could become the primary artifact for human review and the source of record at a time when AI-generated code is outpacing customary code review.

## 1. 论文的核心问题与整体含义（研究动机和背景）
- **研究动机**：随着大型语言模型（LLM）和编码智能体的普及，“vibe coding”（通过自然语言对话让AI生成代码）成为主流。但该模式导致大量AI生成代码，开发者意图仅暂存在聊天记录中，代码晦涩难懂，传统代码审查（code review）难以跟进。AI代码产出速度已超过人工审查能力（DORA 2025报告）。
- **核心问题**：如何从vibe coding会话的对话轨迹和代码产物中，自动提取出**可验证、抽象且强健的自然语言规范（specification）**，以代替代码成为主要审查工件和记录来源。
- **整体含义**：本文提出AfterVibe框架，实现了“事后规范恢复”（retrospective specification recovery），通过再生测试（regeneration test）确保规范忠实于开发者意图，并验证其足够强以让盲AI agent重新生成等价代码。愿景是**规范取代代码**成为AI时代的首要人类创作工件。

## 2. 论文提出的方法论：核心思想、关键技术细节
- **核心思想**：利用LLM从vibe coding会话的对话轨迹（用户指令、agent工具调用、消息）和代码差异（diff）中，蒸馏出**结构化自然语言规范**，并通过**再生测试**循环验证其强度与抽象性。
- **技术流程**（对应图4）：
    1. **规范蒸馏（Spec Distillation）**：使用提取提示（图5）将代码diff和对话轨迹转化为由三部分组成的规范：
        - Part A（规范描述）：意图与原理（Intent and Rationale）、关键设计决策（Essential Design Decisions）、不可发现事实（Undiscoverable Facts）。
        - Part B（需求清单）：以声明性语句列出行为要求，聚焦结果而非实现。
        提示强制抽象：除非必要且不可发现，否则禁止代码、标识符、路径等。
    2. **三级验证流水线（Three-Tier Verifier）**：
        - 第一级：**灵活测试执行**（Flexible Test Execution）——运行从会话中提取的测试命令，允许对接口轻微调整（如重命名入口点）。
        - 第二级：**验证条件检查**（Verification Conditions，VCs）——从对话轨迹、代码diff和人工摘要中提取独立的行为属性（如“日期范围选择器应过滤所有三个图表”），作为无偏 oracle。
        - 第三级：**真实对齐检查**（Ground-Truth Alignment）——LLM agent判断再生代码C′是否与原始代码C在会话意图下行为等价。
    3. **再生测试（Regeneration Test）**：新盲AI agent仅接收规范S，从相同基提交重新实现代码C′。三级验证器给出再生分数（0-6分），定义为：`regen_score = 3*flex + 2*vc + align`（flex为通过灵活测试比例，vc为通过VC比例，align为二元对齐）。
    4. **迭代规范强化（Iterative Spec Refinement）**：对分数低于阈值（默认6.0）的任务，针对每个失败的VC，LLM分析分歧原因并重写完整规范，再经完整流水线重新评估。经过K轮强化后保留每任务最佳规范。
- **公式**（文字描述）：再生测试若验证器输出高于τ则通过，否则失败。强化循环遵循生成-批评-改进范式。

- **RQ3 抽象vs强度**：更具体的规范（diff-preserving）获得更高分（5.55），但62.6%代码标记泄漏，优势主要来自真实对齐。标准策略（5.06，压缩比5.69×，泄漏38.6%）在抽象和强度间取得最佳平衡。
- **RQ4 再生多样性**：通过再生的成对代码间chrF 0.899（接近自身上限0.982，远低于无关对下限0.180），表明规范在约束行为的同时允许结构多样性。

- **启示**：支持“环境基函数假设”——agent从代码库中自行推断大量上下文，规范只需描述不可发现的事实。规范可以取代代码成为审查和记录的主要工件。

## 7. 优点：方法或实验设计上有哪些亮点
- **新颖的问题定义**：首次提出“事后规范恢复”任务，直接针对vibe coding的实际痛点——意图记录缺失。
- **自验证机制**：通过“再生测试”自动评估规范质量，无需人工判断；三级验证器结合执行、检查和对齐，降低单一偏差。
- **抽象性与强度的平衡**：通过模板化提取（区分意图、设计决策、不可发现事实）和提示强制抽象，同时保留必要具体信息。
- **迭代强化**：利用验证条件失败信息循环改进规范，类似于反馈驱动的自动化优化。
- **负控制实验**：精心设计三种退化情况（预变更、部分、错误代码）来证明验证器的判别力，增加了结果可信度。
- **多样性分析**：通过分析再生代码间的相似度量化规范的抽象程度，验证了“约束what而不规定how”。
- **权重敏感性检查**：确保结论对权重配置不敏感。
