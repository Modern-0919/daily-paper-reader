## 摘要
使用工具的LLM智能体大多是在假设所有工具都能工作的情况下进行评估的。当工具超时、返回过时一周的值，或在部署过程中其描述被污染时，开发人员需要一种可控的方式来复现故障、测试修复，并在部署前确认修复有效。我们提出AgentCheck，一个开源Web工作台，它将MCP服务器转变为干预界面。AgentCheck让智能体在其真实工具上运行，记录每个工具的响应，然后使用故障（12种类型）注入器扰动响应后重新运行智能体。匹配的工具调用从缓存中重放，而后续的工具调用在智能体分歧后实时进行。这形成了一个复现-干预-确认循环：开发人员切换缓解措施，针对相同的故障重新运行，查看故障是否消失。评分分为两部分：确定性通过/失败规则，以及用于解释性标签的LLM评判器，并通过人类标注进行验证。在五个智能体中，最佳通过105/120个场景，最弱仅通过77个。失败通常是无声的，自信地使用不正确的工具输出，而非崩溃。在最弱的智能体上，重试缓解措施将超时错误故障的成功率从最低30%提高到100%，而过时数据故障无论采取何种缓解措施，成功率仍维持在3-4/10。AgentCheck使得这些故障模式在部署前具有可复现性、可比性和可验证性。

## Abstract
Tool-using LLM agents are mostly evaluated assuming all tools work. When a tool times out, returns a week-stale value, or has its description poisoned in deployment, the developer needs a controlled way to reproduce the failure, test a fix, and confirm the fix worked before deployment. We present AgentCheck, an open-source web workbench that turns an MCP server into an intervention surface. AgentCheck runs an agent against its real tools and records every tool response, then re-runs the agent with the response perturbed by a fault (12 types) injector. Matching tool calls are replayed from cache, and later tool calls go live after the agent diverges. This yields a reproduce-intervene-confirm loop: the developer toggles a mitigation, re-runs against the identical fault, and sees if the failure goes away. Scoring has two parts: deterministic pass/fail rules, plus an LLM judge for interpretive labels, validated against human annotations. Across five agents, the best passes 105/120 scenarios and the weakest only 77. The failures are usually silent, confident use of incorrect tool outputs rather than crashes. On the weakest agent, a retry mitigation raises success on timeout error faults from as few as 30% of cases to 100%, whereas stale-data faults remain near 3-4 of 10 regardless of the mitigation. AgentCheck makes these failure modes reproducible, comparable, and verifiable before deployment.

当前对工具型LLM智能体的评估大多假设所有工具都能正常工作（如GAIA、SWE-bench、ToolBench等）。然而在实际部署中，工具可能出现超时、HTTP错误、返回过时数据、第三方服务器描述被投毒等情况，导致智能体静默地使用错误输出甚至执行恶意指令。现有工作（如MCPTox、ToolMisuseBench）已揭示这些故障的高成功率，但缺乏一个可控的、可复现的、可验证修复的测试框架。AgentCheck应运而生，旨在为开发者提供在部署前复现故障、测试缓解措施并确认修复效果的工作台。

  - **受控比较**：同一任务下先后执行clean run（记录所有工具真实响应）、faulted run（在注入点p用注入器g改变一个响应，其余从缓存重放，后续调用实时执行）、mitigated run（可选，对智能体应用缓解措施后再次以相同p,g运行）。

### 6. 论文的主要结论与发现
- 智能体性能差距显著：DeepSeek通过105/120，Llama仅77/120。这种性能差距在传统任务成功基准（假设工具正常）中无法体现。
- **弱点集中**于数据质量故障（B类：停滞、矛盾、无关、空结果），所有智能体在此类表现最差（DeepSeek 29/40，Llama 13/40）。这些故障不返回错误码，易被快乐路径基准忽略。
- **静默失败**：失败多为自信使用错误工具输出，而非崩溃或明确报错。
- **缓解效果因故障类型而异**：重试缓解对工具执行故障（A1超时等）提升显著（如Llama在A1上从3/10提升至10/10），但对过时数据（B1-B3）几乎无效（仍在3–4/10左右）；C4数据窃取不受所列缓解措施影响（目标为执行层面，未针对外泄）。
- **“修复确认”循环有效**：在示例中，超时故障通过重试缓解后，所有确定性检查通过并被标记为fix_confirmed。

### 7. 优点
- **方法创新**：首次将MCP服务器作为干预面，实现受控的故障注入与缓存重放，形成可复现–干预–确认闭环。
- **故障类型全面**：12种故障覆盖工具执行、数据质量和安全三大方面，源自现有基准并补充原创。
- **交互式仪表板**：并排可视化clean/fault/mitigated轨迹，直观定位分歧点，降低使用门槛。
- **双层评分**：主要采用确定性检查（无歧义），辅以LLM评判器提供诊断解释（但标为上限而非真理）。
- **开源且可扩展**：发布120个场景、评分工具及所有代码（MIT许可）；支持用户连接自己的MCP服务器和测试自定义智能体。
- **实用性强**：直接回答开发者在部署前最关心的问题：“我的智能体在故障下是否会编造？我的修复能否关闭问题？”

### 8. 不足与局限
- **仅单故障注入**：未考虑复合故障或级联故障（一个故障引发另一个），实际场景更复杂。
- **场景覆盖率有限**：12种故障类型不代表所有可能的失败模式；政策相关或长时推理故障未包含。
- **数据质量场景（B类）为自创**：实验结论可能存在人为偏差，需在其他来源上验证。

- **缓解实验仅针对最弱智能体（Llama）**：未在更强智能体上验证通用性。
- **无算力基准**：缺乏GPU、时间等资源报告，不利于可重复性中的资源对比。
- **隐私与安全**：使用过程中会向LLM提供商发送提示和工具调用，需用户自行确保合规。
