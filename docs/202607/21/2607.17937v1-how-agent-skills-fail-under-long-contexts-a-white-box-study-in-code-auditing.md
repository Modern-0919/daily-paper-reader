## 摘要
智能体技能将程序性指令和检查打包供通用智能体使用，但加载一项技能并不能保证每个需求在长时间的工具使用轨迹中始终保持激活。我们在一个源自生产环境的白盒代码审计工作流中研究这一问题。保持任务和24个工件检查不变，我们改变周围上下文，并分类故障首次出现的位置：需求丢失、编辑漂移、检查失败或非智能体评估器/运行时故障。使用gpt-5.4-mini的Codex在10,991字符的干净上下文中通过了10次中的8次，但在299,140字符的相关上下文和等长度的无关上下文中仅通过了3/10。这一50个百分点的差异很大，但在双尾Fisher检验下仍处于趋势水平（p = 0.0698）。然而，在两种长上下文条件下，需求覆盖率仍保持在92%以上，这表明少数遗漏可能导致原本完整的工件失效。第二个任务通过了所有干净和长上下文运行，因此证据不支持存在通用的上下文长度阈值。详细的内部检查表通过了10/10次运行，而通用自我检查仅为5/10（p = 0.0325）。编码智能体框架可能通过选择更小的工作集来提供帮助，但并不能消除故障。我们没有引入上下文腐烂或新的通用监控方法；我们为白盒代码审计提供了一个有界的故障分类和实证案例研究。

## Abstract
Agent Skills package procedural instructions and checks for use by general-purpose agents, but loading a skill does not guarantee that every requirement remains active throughout a long tool-using trajectory. We study this problem in a production-derived, white-box code-audit workflow. Holding the task and 24 artifact checks fixed, we vary the surrounding context and classify where failures first become visible: lost requirements, editing drift, failed checking, or non-agent evaluator/runtime failures. Codex with gpt-5.4-mini passes 8/10 runs in a 10,991-character clean context but only 3/10 in both a 299,140-character relevant context and an equal-length irrelevant context. This 50-percentage-point difference is large but remains trend-level under two-sided Fisher tests (p = 0.0698). Requirement coverage nevertheless stays above 92% in both long conditions, showing that a few omissions can invalidate an otherwise complete artifact. A second task passes all clean and long runs, so the evidence does not support a universal context-length threshold. A detailed external checklist passes 10/10 runs, compared with 5/10 for a generic self-check (p = 0.0325). Coding-agent scaffolds may help by selecting a smaller working set, but they do not eliminate failures. We do not introduce context rot or a new general monitoring method; we provide a bounded failure classification and empirical case study for white-box code auditing.

### 1. 论文的核心问题与整体含义（研究动机和背景）

- **研究动机**：当前编码智能体（coding agent）依赖可复用的“智能体技能”（Agent Skill）——即打包的程序化指令、引用和检查点。然而，加载技能后，智能体在长时间的工具使用轨迹中并不保证每个需求始终保持激活。常见故障包括：遗漏强制数组、覆盖保护字段、半途而止、忽视工具输出中的矛盾等。此类“窄失败”（narrow miss）在代码审计中会使整个工件失效。
- **背景**：已有研究指出长上下文会导致模型有效上下文缩水（Lost in the Middle、RULER等），但缺乏对智能体技能在长轨迹中具体失效位置的细粒度分析。现有编码智能体基准往往仅报告最终仓库状态，而轨迹研究揭示失败可能早在最终步之前就发生。本研究旨在填补这一空白：在固定技能任务和检查的条件下，变化上下文，观察哪些显式需求会失败，以及简单的外部检查表能否预防失败。
- **整体含义**：本文并非提出新的监控语言或通用失效理论，而是通过白盒实验提供一个透明的案例研究和实用的故障分类，为后续更大规模预注册研究提供证据和设计参考。

### 6. 论文的主要结论与发现

- **RQ1：长上下文降低主任务通过率**。干净上下文8/10通过，相关长和无关长均仅3/10通过（+50个百分点观察差异，Fisher p=0.0698，趋势水平）。需求覆盖率仍很高（>92%），说明失败往往是稀疏的：少量字段缺失导致整体工件不可用。
- **RQ2：语义相关性不区分长上下文效果**。相关与无关长上下文通过率相同（3/10 vs 3/10），Fisher p=1.0。不能得出“相关材料更坏”的简单结论。
- **RQ3：四种可见故障类**：需求丢失（如T002数组缺失）、编辑漂移（如修改保护字段）、检查失败（自检忽略遗漏）、非智能体故障（运行时/评估器错误）。多数失败伴随沉默成功消息（86.4%的历史失败运行声称完成）。
- **RQ4：详细检查表优于通用自检**。在相关长条件下，详细检查表10/10通过，通用自检5/10（Fischer p=0.0325）。机制：详细检查表在完成边界重新引入被遗忘的需求，而自检继承同一盲点。
- **RQ5：编码支架可通过缩小工作集缓解，但不消除失败**。gpt-5.5在小捆绑直接和Codex下成功，但在全直接下失败；gpt-5.4-mini的Direct变体均失败，Codex在干净下成功但在长上下文下仅3/10。
- **跨任务边界**：Revert SL-007在所有8个运行（干净、相关、无关长）中全部通过，表明长上下文效应不是通用的。存在稳定的反例。
- **检查覆盖率与严格任务成功的矛盾**：高覆盖率（>92%）但低严格成功率（30%），强调审计等场景中严格任务成功是必要指标，覆盖率可能误导。

1. **白盒实验设计**：固定任务、检查、工作区，仅变上下文，控制严谨，能直接关联需求到可见失败，无需访问内部推理。
2. **故障分类实用且可操作**：四类故障（需求丢失、编辑漂移、检查失败、非智能体）基于首次可见证据，有明确决策规则，便于后续诊断和干预选择。
3. **区分需求覆盖率与严格任务成功**：揭示高覆盖率下仍可能整体失败，对安全/审计场景非常重要。
4. **公开基础设施可靠性**：单独报告中断尝试，指出50.9%的评分率并分析不平衡，增加了研究的诚实性，避免隐式丢弃不利数据。

7. **透明成本报告**：记录token用量和等成本，允许读者评估研究规模和可复现性。

### 8. 不足与局限：包括实验覆盖、偏差风险、应用限制

- **统计效力不足**：主比较p=0.0698未达α=0.05，样本量小，置信区间宽。扩展策略（观察后增加样本）削弱了推断的确定性。
- **单任务依赖**：主要结果基于一个工作流（Stage 4CD），结论的泛化性存疑。仅一个独立任务（Revert SL-007）作为反例，缺乏独立的正向复制。
- **上下文构建粗糙**：字符匹配不保证token数、语义密度、文件结构等相同，相关与无关材料在可读性、预期搜索行为上可能系统性地不同。
- **模型和支架受限**：主模型仅gpt-5.4-mini+Codex，其他模型探针稀疏，且因网关协议限制无法直接比较Codex与Claude Code。没有评估其他主流智能体框架（如OpenHands等）。
- **基础设施缺失偏倚**：无关长条件中20%评分率可能引入严重选择偏倚——如果容易失败的任务更容易断开，则有效运行通过率被高估。作者承认无法区分。
- **故障分类主观性**：虽有序规则，但依赖人工解读保存的工件和日志，未进行独立双编码验证。分类不能确定因果链（例如，需求丢失可能源于前一步未读取，但分类仅记录首次可见证据）。
- **缓解干预未做跨任务验证**：详细检查表仅在相关长上下文的一个任务上测试，优势可能局限于数组/基数类约束，未在其他审计工作流中验证。
- **外部有效性有限**：代码审计场景具有结构化输出和确定性保护规则，结果可能不适用于浏览器使用、文档编辑、对话式任务等更开放的场景。
- **未区分长上下文的子维度**：未设计因子实验分离长度、语义相似度、活跃义务数量、证据位置等贡献。因此“上下文腐烂”仍是经验性描述而非机制性解释。
