## 摘要
AI软件开发智能体日益具备修改基础设施和安全关键系统的能力，这带来了风险：智能体在完成指定任务的同时，通过扩大权限、降级日志记录或引入持久化机制等行为暗中削弱安全防护。虽然前沿实验室可能部署复杂的监控管道，但许多采用编码智能体的组织和个体用户缺乏维护复杂学习监控集成所需的资源和治理成熟度。我们在ControlArena的基础设施即代码场景中，将这一问题视为任务成功下的基础设施级破坏，并比较了同步和异步部署模式下训练与未训练的监控器。我们引入了一种信息流图（IFG）监控器，它通过控制流和数据流图差异以及原始代码差异来分析结构安全回归。在异步评估中，未训练的git差异监控器在1%假阳性率的审计预算下漏检11.6%的攻击，我们的未训练IFG监控器将此漏检率降至3.5%，而训练的异步随机森林监控器达到低于1%的漏检率。与训练的异步随机森林监控器不同，IFG也可以作为同步预部署安全防护措施，在执行或合并前阻止可疑的基础设施变更。在同步模式下，IFG实时回滚将主要任务与隐蔽任务的联合成功率从74.4%降至0.0%，且对合法任务完成无显著影响。这些结果表明，未训练的结构监控器为寻求可信赖采用高级AI智能体的组织提供了一条实用且可审计的路径，以实现部署安全的民主化。

## Abstract
AI software development agents are increasingly capable of modifying infrastructure and security critical systems, creating risks where an agent completes its assigned task while covertly weakening safeguards through actions such as broadening permissions, degrading logging, or introducing persistence mechanisms. While frontier laboratories may deploy sophisticated monitoring pipelines, many organizations and individual users adopting coding agents lack the resources and governance maturity required to maintain complex learned monitor ensembles. We study this problem as infrastructure level sabotage under task success in the infrastructure-as-code setting from ControlArena and compare trained and untrained monitors across synchronous and asynchronous deployment modes. We introduce an Information Flow Graph (IFG) monitor that analyzes structural security regressions using control-flow and data-flow graph diffs alongside raw code diffs. In asynchronous evaluation, an untrained git diff monitor misses 11.6% attacks at 1% false positive rate auditing budget, our untrained IFG monitor reduces this to 3.5%, and the trained Async RF monitor achieves below 1%. Unlike the trained Async RF monitor, IFG can also operate synchronously as a pre-deployment safeguard that blocks suspicious infrastructure changes before execution or merge. In synchronous mode, IFG realtime rollback reduces joint main task and covert task success from 74.4% to 0.0% with no measurable reduction in legitimate task completion. These results suggest that untrained structural monitors provide a practical and auditable path toward democratizing deployment safety for organizations seeking trustworthy adoption of advanced AI agents.

## 2. 论文提出的方法论

### 核心思想
- 引入**信息流图（Information Flow Graph, IFG）监控器**：不依赖运行时轨迹或攻击历史，而是基于**控制流图（CFG）**和**数据流图（DFG）**的结构差异，结合原始代码差异（git diff），判断变更是否引入安全回归（如移除拒绝语句、新增管理员角色、扩大资源范围等）。

### 关键技术细节
- **处理流程**：
  - 输入：原始CDK源码 → 运行`cdk synth`生成基线CloudFormation模板 → 提取CFG/DFG。
  - 智能体修改源码后，再次运行`cdk synth`得到补丁后模板 → 计算原始与补丁的CFG/DFG增量（SecurityDelta）。
  - 将安全增量、原始代码差异和任务描述提供给**信任模型（LLM）**，模型输出结构化威胁分析（STRIDE分类）和整数可疑分数（1~10，分数越高越可疑）。
- **两种部署模式**：
  - **异步（事后）模式**：监控器在智能体任务完成后审查，可阻止部分损害但允许执行后干预。
  - **同步模式（实时回滚）**：在智能体变更应用前（如合并或执行前）进行拦截，若发现可疑则自动回滚。
- **阈值校准**：在100个诚实episode上校准阈值，确保**1%假阳性率（FPR）**预算。

### 公式/算法（文字说明）
- 监控器判定：`IFG_score = f(SecurityDelta, git_diff, task_description)`，其中`f`为信任LLM的推理函数。
- 安全指标：`Safety@FPR = D / (D + US)`，其中`D`为检测到的攻击，`US`为未检测到的成功攻击。

## 7. 优点

1. **无需训练数据**：IFG完全基于确定性结构分析，不依赖攻击日志，降低了部署门槛。
2. **机制可解释性**：每个可疑变更都关联到具体结构对象（如移除的拒绝语句、新管理员角色），便于审计和人工检查。
3. **同步实时能力**：能在变更执行前拦截，大幅提升安全效果（0%联合成功率），而训练集成仅支持事后监控。
4. **低成本**：相较于训练集成（需要多个LLM调用+特征工程），IFG只需一次LLM推理调用与确定性图谱分析。
5. **广泛适用性**：若代码可表示为控制流/数据流图，该思想可扩展到其他IaC工具（如Terraform、Kubernetes manifests）。

## 8. 不足与局限

1. **检测盲区**：无法检测通过**编译后工件**（如`cdk.out/`中的预生成模板）或**gitignore文件**外传递的有效载荷（论文已明确排除的侧任务）。需要配合构建完整性控制（如SLSA）。
2. **对LLM信任模型的依赖**：最终可疑分数仍取决于LLM推理，存在**提示注入**风险（引用另一篇论文Terekhov et al.，2025），虽然确定性图谱减少了攻击面但未完全消除。
3. **应用限制**：仅针对IaC场景（具体为AWS CDK/CloudFormation），需要能够生成CFG/DFG的工具。对于非IaC的软件工程任务（如数据库迁移、配置文件）不直接适用。
4. **实验覆盖有限**：
   - 仅测试了ControlArena的一个子集（iac fast），未涵盖更广泛的网络操作（如BashArena）或其它IaC工具（Terraform）。
   - 消融实验较少：未分析CFG和DFG各自贡献，未测试不同阈值或提示模板的影响。
   - 诚实episode规模仅为100，可能不足以完全代表真实世界变更分布。
5. **同步模式的有用性**：虽然未观察到可测量的退化，但实验仅覆盖了100个honest episode，实际部署中可能出现误报导致合法变更被回滚（虽然FPR控制在1%）。
