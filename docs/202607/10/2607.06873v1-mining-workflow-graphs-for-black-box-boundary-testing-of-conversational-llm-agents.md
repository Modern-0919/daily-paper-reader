## 摘要
会话式LLM智能体的内部工作流出现故障时（例如未确认即完成交易）可能造成现实世界的危害。测试这些状态相关的故障非常困难，因为关键边界（如身份检查与确认门）隐藏在多轮会话的前提条件之后，使得标准测试无法触及。我们提出AgentEval，一种发现并强化这些有状态边界的黑盒测试框架。AgentEval与智能体交互，挖掘其行为的"会话工作流图"。AgentEval并非盲目提示，而是利用该图的结构将特定的守卫和前提条件枚举为测试目标，在施加扰动之前回放通向边界的对话路径。随后，AgentEval执行每个测试，仅通过对话轮次判断其通过或失败。我们将AgentEval与能够访问智能体底层源代码的特权白盒审计器进行基准测试（AgentEval从未见过该代码）。在四个τ^3-bench智能体上，AgentEval成功生成了覆盖每个智能体23至38个不同边界的测试；消融研究将增益归因于图的结构：相比仅使用提示的基线，其不同边界数量从12个提升至23个，且重复率和误报率更低。

## Abstract
Conversational LLM agents can cause real-world harm when their internal workflows fail, such as completing a transaction without confirmation. Testing these state-dependent failures is difficult because critical boundaries, such as identity checks and confirmation gates, are hidden behind multi-turn conversational prerequisites, rendering them inaccessible to standard tests. We present AgentEval, a black-box testing framework that discovers and stresses these stateful boundaries. AgentEval interacts with an agent to mine a \emph{conversational workflow graph}, a model of its behavior. Instead of prompting blindly, AgentEval uses this graph's structure to enumerate specific guards and prerequisites as test targets, replaying the conversational path to a boundary before applying a perturbation. AgentEval then executes each test, determining whether it passes or fails using only the conversation turns. We benchmark AgentEval against a privileged, white-box auditor with access to the agent's underlying source code, which AgentEval never sees. On four $τ^3$-bench agents, AgentEval successfully generates tests covering $23$--$38$ distinct boundaries per agent; ablation studies attribute the gain to the graph's structure: $23$ distinct boundaries versus $12$ with a prompt-only baseline, at lower duplicate and false-alarm rates.

## 1. 核心问题与整体含义（研究动机和背景）

- **核心问题**：会话式大语言模型（LLM）智能体在实际部署中常因内部工作流故障造成严重后果（如未确认即取消预订、对未验证用户执行操作等）。这类故障发生在**工作流边界**（如确认门、资格检查、值验证等）被绕过时。然而，这些边界被隐藏在多轮对话前提条件之后，且智能体通常作为黑盒存在（只能通过对话接口访问），导致标准测试方法无法有效发现此类状态依赖的缺陷。
- **整体含义**：需要一种黑盒测试框架，能够自动发现智能体中隐藏的边界，构造能够到达边界的对话路径，并施加扰动以检验智能体是否违反预期行为，从而实现对已部署智能体的回归测试。

## 2. 方法论：核心思想、关键技术细节

- **核心思想**：通过黑盒对话交互挖掘智能体的**会话工作流图**（Conversational Workflow Graph），利用图的结构将边界枚举为测试目标，沿路径回放到目标状态后施扰，仅凭对话文本判定结果。
- **关键技术细节**：
  - **两阶段框架（AgentEval）**：
    1. **发现阶段（Discovery）**：
       - 探索：采用**阶段引导的探索**（Phase-Guided Discovery），包括能力发现、快乐路径、一致性检查三种探索阶段，由LLM驱动的规划器决定探索目标，驱动执行器与智能体对话，收集多轮对话轨迹。
       - 事件抽象：LLM将每轮对话抽象为用户动作（user action）和智能体活动（agent activity），不同措辞但含义相同的回复映射到同一活动标签。
       - 图构建：采用**直接跟随图**（Directly-Follows Graph）方法，统计轨迹中活动间的顺序关系，构建节点（活动）、边（过渡）、入口/出口节点。边携带频率和驱动该过渡的用户动作。
       - 测试合成：直接成功轨迹生成**功能测试**；基于图结构枚举节点/边/入口/出口作为边界目标，由LLM评分边界潜力后保留前N个，并生成**边界测试**（目标位置、预期行为、禁止行为）。
    2. **执行阶段（Execution）**：
       - LLM驱动的执行器根据测试目标逐轮生成用户消息，与智能体交互到给定预算。
       - 独立的LLM**裁判**（Judge）读取完整对话轨迹，对照测试的成功/失败标准给出通过、失败或不确定的判决。
  - **核心公式与算法**：算法1（构建图）：遍历每条轨迹，累加节点支持度、边频率和用户动作，记录入口和出口。算法2（图引导边界测试合成）：枚举所有结构位置，按LLM评分排名，生成不超过预算的边界测试。
  - **关键创新**：利用图的结构性位置驱动测试生成，而非仅依赖自由提示，从而覆盖更多样化的边界，避免重复。

## 7. 优点

- **完全黑盒**：仅依赖对话接口，无需内部源码、日志或规范，适用于已部署系统。
- **图引导的结构化测试**：克服了自由提示容易重复少数流畅路径的局限，系统性覆盖所有结构位置，生成低重复、高多样性的边界测试。
- **分阶段设计**：发现阶段生成的可重用测试计划可用于回归测试；执行裁判独立于生成器，判决仅凭可见对话，保证可复现性。
- **基准设计创新**：引入特权审计器，基于源码提供地面真实性，评估测试计划有效性和执行结果，支撑公平比较。
- **跨域泛化性**：在四个不同业务域上一致有效，显示方法通用性。

## 8. 不足与局限

- **依赖可见行为**：只能测试在对话中可见的故障；后端状态错误但回复正确的故障无法检测（论文承认此类问题）。
- **需要重复交互模式**：该方法假设智能体有规律的工作流（如路由、槽填充、确认门等）；对于完全开放、不可重复的智能体，挖掘的图可能很薄，边界目标少。
- **单次运行敏感性**：论文中所有结果基于单次实验，未报告统计显著性（如多次运行的标准差），小差异可能不可靠。
- **边界测试有效性率受限**：在航空和零售域，边界测试有效性率较低（0.78/0.79），主要由于模型未能一致地将边界场景与正确客户身份绑定，导致测试不可达。这属于合成阶段的技术限制。
- **审计器误差风险**：虽然验证了覆盖审计与专家的一致性，但执行阶段标签（用于计算误报率等）未经独立人工验证，可能存在残余误判。
- **成本**：虽然效率尚可，但整体仍需数百次LLM调用和数小时，对于大规模持续集成可能仍偏高。
