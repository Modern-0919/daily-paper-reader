## 摘要
在长周期LLM系统中，智能体自我进化主要是程序性的：有用的经验不仅是存储的信息，更是用于搜索、调试和验证的可重用程序。然而，当前的评估并未孤立这种形式的迁移。智能体基准测试测试单轮任务解决；记忆基准测试针对信息保留而非程序重用。我们提出了EvoAgentBench，这是一个通过能力引导迁移实现智能体自我进化的基准测试，涵盖四个智能体领域：网络研究、算法推理、软件工程和知识工作。EvoAgentBench从智能体执行中提取基于轨迹的能力，将其规范化到操作单元，并构建特定领域的能力图，链接具有程序重叠的任务。通过设计，每个测试任务都有经过验证的训练侧能力支持。在528/267的训练/测试分割、两种框架和三种骨干模型上，精心策划的能力内容在不同模型家族间可靠迁移，但当前没有自动方法在所有设置中保持正增益。EvoAgentBench将自我进化评估从总体准确率比较转向对经验编码、路由和吸收的细粒度诊断。该基准测试公开于https://huggingface.co/datasets/EverMind-AI/EvoAgentBench。

## Abstract
Agent self-evolution in long-horizon LLM systems is largely procedural: useful experience is not merely stored information, but reusable procedures for searching, debugging, and verification. Yet current evaluations do not isolate this form of transfer. Agent benchmarks test single-episode task solving; memory benchmarks target information retention rather than procedural reuse. We introduce EvoAgentBench, a benchmark for agent self-evolution via Ability-guided transfer across four agentic domains: web research, algorithmic reasoning, software engineering, and knowledge work. EvoAgentBench extracts trace-grounded Abilities from agent executions, canonicalizes them into operational units, and builds domain-specific Ability Graphs linking tasks that share procedural overlap. By design, every test task is backed by verified training-side Ability support. Across a 528/267 train/test split, two scaffolds, and three backbones, curated Ability content transfers reliably across model families, but no current automatic method sustains positive gain in all settings. EvoAgentBench shifts self-evolution evaluation from aggregate accuracy comparison to fine-grained diagnosis of experience encoding, routing, and uptake. The benchmark is publicly available at https://huggingface.co/datasets/EverMind-AI/EvoAgentBench.

### EvoAgentBench：通过能力迁移实现智能体自我进化的基准测试

#### 1. 论文的核心问题与整体含义（研究动机和背景）

- **核心问题**：当前评估智能体自我进化（Agent Self-Evolution）的基准存在一个共同的盲区。它们主要评估单轮任务解决能力或信息保留能力，但未能有效衡量核心能力：**从过去的执行轨迹中提取可重用的程序性知识（如搜索策略、调试流程），并将其迁移到新的、不同的任务中**。
- **研究背景**：虽然已有多种方法让智能体通过外部工件（如技能库、案例、工作流）来改进自身，但缺乏一个专门的评估框架来隔离和衡量“轨迹到程序”（trajectory-to-procedure）的迁移能力。现有基准要么混淆了工件质量与任务顺序、检索策略等因素，要么仅在一个任务内评估程序质量，而非跨任务的程序迁移。
- **整体含义**：论文提出，一个用于评估智能体自我进化的基准，必须满足一个结构性原则：**能力支持但实例不重叠** (ability-supported yet instance-disjoint)。即，测试任务必须与训练任务共享可迁移的程序性能力，但具体实例不同，以确保改进源于程序性知识的迁移，而非记忆答案或近似实例。

#### 2. 论文提出的方法论：核心思想、关键技术细节

- **核心思想**：将智能体自我进化评估形式化为一个**受控的经验到迁移的评估问题**。通过构建一个精心设计的评估平台，使得不同方法之间的差异仅在于它们如何编码和重用训练经验。
- **关键技术：能力图（Ability Graph）构建** 这是一个三阶段流水线：
    - **阶段1：多骨干轨迹收集**。在无技能（no-skill）设置下，使用多个不同的构造骨干模型（如 Kimi-K2.5, GLM-5.1, DeepSeek-V3.2）在源任务上执行，获取丰富的成功和失败的执行轨迹。
    - **阶段2：轨迹基础的能力提取与规范化**。
        - 首先，从每个任务的轨迹中提取出原始的“能力卡”（Ability Cards），每个卡包含触发条件 (`γa`)、可重用程序 (`πa`)、证据、边界 (`∂a`) 和角色（方法、守卫、工作流）。
        - 然后，通过嵌入相似度、LLM裁决和专家评审，将不同任务中功能等价的原始能力卡合并成规范的“能力单元”（Canonical Ability Units）。这些单元构成了定义迁移路径的基本词汇。
    - **阶段3：能力引导的数据划分**。
        - 基于规范的能力单元构建无向图（Ability Graph），如果两个任务共享至少一个能力单元，则它们之间存在边。
        - 使用 Louvain 社区检测算法将图划分为能力社区。
        - 在分割训练/测试集时，施加一个关键约束：**每个测试任务必须与至少一个训练任务共享至少一个边-合格的能力单元**。
        - 这保证了测试集中每个任务都能从训练集中获得“程序性支持”，从而实现了“能力支持但实例不重叠”的核心原则。

#### 3. 实验设计：数据集、Benchmark、对比方法

- **数据集与场景**：基准测试涵盖四个长期、智能体任务领域，每个领域由一个源基准实例化：
    - **网络研究** (Web Research)：BrowseComp-Plus
    - **算法推理** (Algorithmic Reasoning)：LiveCodeBench
    - **软件工程** (Software Engineering)：SWE-Bench Verified
    - **知识工作** (Knowledge Work)：GDPVal
- **Benchmark**：EvoAgentBench 本身。它提供了一个 528 / 267 的训练/测试分割（总计 795 个任务），并构建了包含 170 个规范能力单元的 Ability Graph 作为诊断参考。
- **对比方法**：
    - **基线 (Vanilla)**：无进化状态。
    - **自动进化方法**：
        - **Memento**：检索最相似的训练案例（基于查询嵌入）。
        - **ReasoningBank (RB)**：从经验中蒸馏自然语言策略，并根据策略相似度检索。
        - **GEPA**：从训练数据中进化出一个单一的提示，广播至所有测试任务（无检索）。
    - **诊断参考 (Anchor Skill)**：使用论文提取流水线生成的、由能力图（Ability Graph）路由的人工策划技能。这并非一个可部署的方法，而是用于衡量能力内容的可迁移性上限。其构造骨干与评估骨干不同。

- 论文中**未明确说明**所使用的算力资源（GPU 型号、数量、训练时长等）。

#### 5. 实验数量与充分性

- **实验数量**：实验规模相当充分。在 2 种智能体框架（OpenClaw, Nanobot）与 3 种评估骨干模型（Qwen3.5-27B, Qwen3.5-397B, Gemma-4-31B）组合成的 6 个设置中，测试了 4 种不同的进化条件。每个结果均为 3 次独立运行的平均值，并报告了标准误差。
- **充分性与公平性**：
    - 实验设计较为全面，覆盖了不同框架、不同规模/家族的模型。
    - **公平性**：文中明确指出，在每个 agent-骨干组合中，任务、工具集、超时、评分合同和基础agent配置都是固定的。方法间的唯一差异在于进化状态的构建和应用方式。这有效地控制了变量，使比较相对公平。
    - **诊断深度**：实验不仅报告了平均性能，还分析了不同任务类型（已解决 vs. 未解决）的成本变化，并深入探讨了每种方法范式失败的原因，这增加了分析的深度。

#### 6. 论文的主要结论与发现

- **能力内容是跨模型族可迁移的**：由人工策划的参考技能（Anchor Skill）在所有骨干、所有设置、所有领域上都带来了**正收益**（Positive Δ），证明了通过轨迹提取的能力在理论上是完全可以迁移的。
- **当前自动进化方法存在严重的负迁移问题**：所有自动方法（Memento, RB, GEPA）都有至少一个设置（scaffold-骨干-领域组合）出现性能下降（负收益），表现出**脆弱性**和不可靠性。
- **差距在于自动方法的“路由”机制**：由于所有方法共享相同的实验环境，参考技能（Anchor Skill）与自动方法之间的差距，明确指向了当前方法在**经验编码、索引和路由**方面的瓶颈。
- **成本与准确性并非正相关**：进化并不总能降低成本。有些方法即使在成功任务上也会增加交互轮次（如Memento），说明引入不当的工件会带来额外开销。成本-准确性解耦，需要联合报告。

#### 7. 优点：方法或实验设计上的亮点

- **核心创新（能力图）**：提出了“能力支持的转移评估”这一结构原则，并据此构建了能力图。这是论文最核心的贡献，它精准地衡量了“程序性知识迁移”，而非一般的任务表现或记忆。
- **诊断性评估框架**：EvoAgentBench 不仅是一个基准，更是一个诊断平台。它通过提供“能力-支持”的测试环境和参考技能，使得研究者可以精确定位进化失败的具体环节（编码、路由还是吸收），而不仅仅是报告一个分数。
- **严谨的实验控制**：严格控制了除进化状态外的所有变量，使得比较非常干净。这种设置清晰地揭示了不同方法在可迁移知识利用上的真实能力。
- **多维度分析**：不仅关注准确率，还分析了成本变化和不同任务类型（已解决/未解决）上的表现，提供了更全面的视角。

#### 8. 不足与局限

- **领域覆盖有限**：实验仅涵盖了四个**文本**领域的智能体任务。虽然具有代表性，但尚未扩展到多模态、具身智能体等更广泛的场景。
- **构造骨干的依赖性**：能力提取步骤仅使用了一个LLM（Claude Sonnet 4.6），结果可能受此模型能力限制。虽然通过多方判定进行缓解，但不同提取器对最终能力词汇的影响尚未充分研究。
- **评估规模偏小**：由于每个领域的测试集大小在56到86个任务之间，对于更细粒度的领域级分析，可能规模尚显不足。
- **数据污染风险**：论文承认，评估骨干模型可能在预训练阶段遇到过源基准数据，这可能会影响绝对分数，但论文声称由于所有条件共享相同的数据，因此条件间的比较不受影响。然而，潜在的污染仍然是需要考虑的局限。
- **头房偏向**：测试任务被优先选择在构造骨干模型还有提升空间的地方（头房），这意味着基准的“平均增益幅度”可能高于随机抽取任务时的期望值，但方法间的比较仍然是有效的。
