## 摘要
软件智能体基准测试通常报告智能体是否解决了任务，但智能体通过一个框架（harness）达成该结果，该框架控制其看到的内容、可采取的行动、修复哪些故障、验证哪些状态以及记录哪些证据。我们表明，即使任务、环境和基础LLM固定，该框架也能改变智能体的多步骤信念。我们引入了一种信念展开（belief-rollout）诊断方法，该方法在替代框架下，针对进度、风险、可恢复性、约束、失败模式、不确定性、未来成功、修复成本和下一步行动，引出结构化的K步轨迹。我们定义了一个跨框架信念分歧，并将其分解为用于即时界面转移的到达项和用于依赖于时间范围的信念变化的增长项。在受控编码任务和公共基准压力测试中，受阻的行动、压缩的修复、选择性验证和成本感知的证据修剪通常在保持终端成功的同时，改变驱动后续决策的信念。我们进一步引入了BIWM，一种无训练协议，该协议规范化观察结果、记录审查分支、扩展修复轨迹、记录验证掩码、在影子中执行风险分支，并跨框架视图对齐信念轨迹。结果表明，框架设计是智能体评估中的一个实验变量，而非实现细节。我们的代码可在https://github.com/Hik289/Harness-induce-bias.git获取。

## Abstract
Software-agent benchmarks usually report whether an agent solves a task, but the agent reaches that outcome through a harness that controls what it sees, which actions it can take, which failures are repaired, which states are verified, and which evidence is logged. We show that this harness can change the agent's multi-step beliefs even when the task, environment, and base LLM are fixed. We introduce a belief-rollout diagnostic that elicits structured K-step trajectories over progress, risk, recoverability, constraints, failure mode, uncertainty, future success, repair cost, and next action under alternative harnesses. We define a cross-harness belief divergence and decompose it into an arrival term for immediate interface shifts and a growth term for horizon-dependent belief changes. On controlled coding tasks and public-benchmark stress tests, blocked actions, compressed repairs, selective verification, and cost-aware evidence pruning often preserve terminal success while changing the beliefs that drive later decisions. We further introduce BIWM, a no-training protocol that canonicalizes observations, logs censored branches, expands repair traces, records verification masks, executes risky branches in shadow, and aligns belief trajectories across harness views. The results suggest that harness design is an experimental variable in agent evaluation, not an implementation detail. Our code is available at https://github.com/Hik289/Harness-induce-bias.git.

# 多步骤LLM智能体中由框架引发的信念分歧的测量：论文总结

## 1. 核心问题与整体含义（研究动机和背景）

- **核心问题**：在软件智能体评估中，用于控制智能体感知、行动、修复、验证和记录的执行框架（harness）是否会在保持任务、环境和基础LLM不变的情况下，系统地改变智能体的多步信念轨迹（belief trajectory），从而影响后续决策？
- **背景**：现有智能体基准（如SWE-bench、WebArena、OSWorld）几乎只报告最终成功标签，将harness视为中性基础设施，忽略了它如何塑造智能体的内部信念。实际上，不同harness（如阻塞危险动作、压缩修复反馈、选择性验证）会改变智能体对进度、风险、可恢复性、失败模式的估计，而这些中间信念驱动后续行动。
- **整体含义**：Harness设计是智能体评估中的一个关键实验变量，而非实现细节。忽视其影响可能导致对不同系统的比较产生偏差，并隐藏安全风险（如阻塞动作后，智能体仍可能认为危险分支不存在）。

## 2. 方法论：核心思想、关键技术细节

### 核心思想
- 定义并测量**跨harness的信念分歧**（belief divergence）：固定任务、环境和基础LLM，仅改变harness配置，对比由此产生的结构化的K步信念轨迹。
- 将总体分歧分解为**到达项**（arrival term，由即时界面差异引起，如行动和约束表示变化）和**增长项**（growth term，由时域依赖变化引起，如风险、失败模式、概率预测）。

### 关键技术细节

#### Harness六元组定义
每个harness由六个映射构成：观察映射 $O_H$、行动接口 $A_H$、验证器 $V_H$、风险门 $G_H$、修复策略 $R_H$、日志策略 $L_H$。原始参考harness $H_0$ 暴露所有原始信息。

#### K步信念展开
- 信念空间 $\mathcal{B}$ 包含分类字段（进度、风险、可恢复性）、失败模式、约束集、数值字段（不确定性、成功概率等）和推荐动作。
- 迭代过程：从任务和初始历史生成初始信念 $b_0$；提取推荐动作；根据harness中介的上下文生成下一信念 $b_{t+1}$；更新历史。

#### 信念散度 $\mathcal{D}_{\text{belief}}$
- 使用五个子距离的加权和：分类距离、失败模式距离（指示符）、约束集距离（Jaccard变体）、数值距离（归一化绝对差）、行动距离（前8个tokens比较）。
- 权重向量固定为 $(0.30, 0.15, 0.25, 0.25, 0.05)^\top$。

#### 到达-增长分解
- $\mathcal{D}_{\text{arrival}}$ 由约束集和行动距离构成（权重和0.30），反映即时界面差异。
- $\mathcal{D}_{\text{growth}}$ 由分类、失败模式、数值距离构成（权重和0.70），反映时域演化信念。
- $\mathcal{D}_{\text{belief}} = 0.30 \mathcal{D}_{\text{arrival}} + 0.70 \mathcal{D}_{\text{growth}}$。

#### BIWM协议（无训练）
- 规范化观察（canonicalisation）、记录受阻分支日志、展开修复轨迹、记录验证掩码、影子执行风险分支、跨harness对齐（通过投票和平均）。
- 目的：使被harness压缩或隐藏的信息变得可审计，减少对单一harness视图的依赖。

## 6. 主要结论与发现

1. **Harness改变信念轨迹**：即使固定任务、环境和LLM，不同harness（特别是结构化解析、风险门控、修复重型、验证选择性、成本感知）产生显著不同的信念展开，尤其在增长项上。
2. **到达项快速饱和**：行动和约束表示的差异在第一步就接近最大值，掩盖了后续的信念变化。**增长项**（风险、失败模式、概率预测）随horizon持续变化，是更敏感的指标。
3. **长时域下总体分歧稳定但成分持续漂移**：$\mathcal{D}_{\text{belief}}$ 在早期即达平台，但数值预测（如成功概率）在$K=16$到$20$仍变化；失败标签在$K=5$时暂时收敛后又分离。
4. **风险门控导致的信念转变**：阻塞危险行动后，模型将失败归因于策略违规而非行动本身，可能改变风险感知和后续行动推荐。
5. **BIWM暴露隐藏信息**：通过记录受阻分支、展开修复、影子执行等，使被压缩的证据可审计；跨harness对齐可减少单视图依赖。
6. **增长分歧预测行动分歧**：$\mathcal{D}_{\text{growth}}$ 高的样本更可能产生不同的动作分类。
7. **UnsafeRetryRate 高达70%**：在Terminal-Bench危险命令组中，风险门控虽然阻塞行动，但模型在接下来三步内推荐同类危险动作的概率高达0.7，说明阻塞未消除意图。

## 7. 优点

- **问题新颖且重要**：首次系统测量harness对智能体中间信念的影响，挑战了评估中“框架中性”的隐含假设。
- **方法论严谨**：形式化定义harness六元组和信念空间；提出$\mathcal{D}_{\text{arrival}}/\mathcal{D}_{\text{growth}}$分解，揭示被标量指标掩盖的时域依赖变化；提供理论性质（有界性、对称性、单调性条件）。
- **实验设计全面**：受控机制验证+公共基准压力测试+长时域+组件消融+跨基准转移测试；多维度交叉（harness类型、horizon、任务领域、BIWM组件）。
- **实用性**：BIWM为无训练协议，直接应用于现有LLM的推理轨迹，可集成到评估管道中；代码和日志公开，可复现。
- **安全启发**：发现风险门控后UnsafeRetryRate过高，提示安全机制需要与信念更新结合（例如通过影子执行展示被阻塞的后果）。
- **开源**：代码仓库已提供。

## 8. 不足与局限

1. **实验规模较小**：受控任务仅8个，公共基准使用切片（SWE-bench约60-612 runs，Terminal-Bench 60-540 runs），未覆盖全部SWE-bench或更大基准，结论的统计泛化性有限。
2. **基础LLM单一**：所有实验使用同一种LLM（文中未明确具体型号，但推测为某个通用模型），未测试不同模型家族或尺寸下的鲁棒性。
3. **BIWM效果与任务耦合**：某些组件（如展开修复）在Terminal-Bench上效果很小，表明信息损失程度取决于任务类型；BIWM不一定能完全恢复所有被压缩信息。
4. **自洽性分析（Appendix C）** 只是探索性，未严格校准到环境真实结果；预测与最终信念之间的一致性升高不代表正确性提高。
5. **未考虑harness与行动策略的交互**：实验仅静态测量信念差异，未实际执行后续动作并观察闭环结果（除了UnsafeRetryRate初步分析）。
6. **计算开销**：BIWM需要记录额外日志、影子执行等，可能增加评估复杂度。作者未量化额外成本。
7. **安全性双刃剑**：披露信念测量方法可能被滥用，如故意选择使不安全信号消退的harness参数。
