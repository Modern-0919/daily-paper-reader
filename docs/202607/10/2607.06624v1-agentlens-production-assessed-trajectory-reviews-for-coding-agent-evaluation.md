## 摘要
我们提出AgentLens，一个面向交互式代码智能体的生产评估基准。大多数代码智能体基准将一次运行简化为单个比特——任务是否通过？但实际使用这些智能体的人会体验整个轨迹：智能体如何遵循指令、使用工具、验证自身工作、从错误中恢复以及在此过程中与用户交流。AgentLens评估整个轨迹。它将存在客观检查的形式化验证与LLM编写的轨迹回顾及并排比较相结合，使得每次运行都能生成关于得分原因的易读解释。这使得AgentLens不仅可用于模型排序：我们用它来诊断模型行为、比较自身智能体的连续版本，并在夜间评估流程中捕获产品回归。我们将基准作为开源发布，地址为https://github.com/agent-lens/agent-lens-bench。

## Abstract
We present AgentLens, a production-assessed benchmark for interactive code agents. Most code-agent benchmarks reduce a run to a single bit -- did the task pass? -- but the people who actually use these agents experience the entire trajectory: how the agent follows instructions, uses its tools, verifies its own work, recovers from mistakes, and talks to them along the way. AgentLens evaluates that whole trajectory. It pairs formal verification, where an objective check exists, with LLM-written trajectory reviews and side-by-side comparisons, so that each run yields a readable explanation of why the score is what it is. This makes AgentLens useful for more than ranking models: we use it to diagnose model behavior, compare successive versions of our own agent, and catch product regressions in a nightly evaluation pipeline. We release the benchmark as open source at https://github.com/agent-lens/agent-lens-bench.

## 1. 论文的核心问题与整体含义（研究动机和背景）

- **核心问题**：现有代码智能体（coding agent）基准（如SWE-bench、Terminal-Bench）仅用一个二进制信号（通过/失败）来评价一次运行，这忽略了用户在实际使用中体验到的完整交互轨迹。一个通过测试的智能体可能仍然不可靠、过度自信、交互体验差或偏离用户真实需求；而一个失败的智能体可能在过程中做了很多有用的事（如正确定位错误、安全使用工具、优雅恢复）。此外，许多任务没有天然的二元结果（如编写项目文档），单一通过标志无法反映文档的准确性、全面性、与代码的契合度等。
- **整体含义**：AgentLens 旨在填补这一空白，提供一种**生产评估式**的基准，不仅关注最终状态，而是评估整个交互轨迹，从而更贴近用户对代码智能体的真实感知。它结合形式化验证与 LLM 撰写的轨迹审查及并排比较，生成带有证据链接的可读评分解释，用于**诊断模型行为、比较版本、检测产品回归**。

## 2. 论文提出的方法论：核心思想、关键技术细节、公式或算法流程

- **核心思想**：以**完整轨迹**作为评估对象。轨迹包括会话中所有消息（模拟用户消息、智能体回复、工具调用、文件编辑、命令执行、验证尝试、最终应答和最终仓库状态）。通过这一记录来评估质量，而不是仅看最后状态。
- **关键技术细节**：
  - **任务集与模拟用户**：16 个编码智能体场景（16个任务情景），每个场景与默认人格和有毒人格两种模拟用户配对，共 32 条轨迹。任务来自开发者访谈和生产数据（匿名化聚类）。
  - **形式化验证**：提供客观检查，包括仓库状态检查（如 NoChangesVerifier、ExactFileMatchVerifier）、正则表达式检查、测试执行、命令输出检查、静态分析验证器等（表1列出多种 verifier）。仅当所有 verifier 通过才算正式验证通过。
  - **LLM 裁判指标**：每个轨迹由 LLM 裁判按五个维度评分并附书面审查：
    - **End Result**：最终结果的完整性和适用性。
    - **Instruction Compliance**：是否遵循用户的明确规则、步骤顺序和元指令。
    - **Pitfalls**：可修复的行为失败（工具滥用、非生产性循环、过早完成、缺失验证等）。
    - **Pleasantness**：交互质量（清晰、准确、简洁、有帮助，感觉富有成效）。
    - **Tool Calls**：工具使用质量（工具选择、参数正确性、成功失败、从工具错误中恢复、高效与无效调用）。
    每个维度的评分为序数尺度（如 Pitfalls 为 0/0.5/1），并附带结构化证据行（Aspect | Severity | Evidence）。
  - **质量指数 (QI)**：当前为五个 LLM 裁判指标加上形式验证的未加权平均值：Q(a) = (1/|M|) * Σ s_m(a)。这是一个简单代理，直觉上可靠的智能体得分高。
  - **并排比较 (Side-by-Side)**：对同一场景的两个轨迹，使用另一个 LLM 裁判比较它们在单一维度上的表现。裁判接收两个轨迹和事先计算好的单轨迹审查，输出比较性解释和结构化证据行，赋予整数分数 [-5, 5]。
  - **生产管道**：CI 管道自动运行基准，生成轨迹、形式验证结果、指标审查、聚合分数，并通过并排比较与锚点运行对比，标记统计显著的变化作为回归并通知维护者。
- **公式/算法流程**：质量指数公式 Q(a) = (1/|M|) * Σ s_m(a)，其中 M = {End, Instr., Pitfalls, Pleasant., Tools, Formal}。并排评分基于 pitfall burden scores（严重度×频率）计算 delta，映射到整数分数。

- **AgentLens 能够揭示二进制成功信号之外的重要差异**：例如，Gemini 3.1 Pro Preview 模型能力很强，但代理循环可靠性低（skip steps, premature edits, false validation claims）；DeepSeek V4 Pro 和 Flash 的 QI 接近，但策略不同（Pro 更严谨但慢，Flash 更快但松散）；Kimi K2.6 的垫底主要是因为 OpenRouter 侧的工具参数解析 bug 而非模型能力。
- **排行榜显示不同模型各有优劣**：Opus 4.7 在 EAA 框架上综合表现最好（QI=81.5），指令遵循分低于最终结果分；GPT-5.5 在流程合规上更好，但语义可靠性稍弱；Sonnet 4.6 形式验证率高，但愉悦度和工具使用分中等。

## 7. 优点：方法或实验设计上的亮点

- **轨迹级评估**：超越简单二进制通过/失败，评估整个用户体验，包括指令遵循、工具使用、验证行为、错误恢复、沟通质量等，符合开发者实际感知。
- **可解释性**：每个分数都附带由 LLM 裁判撰写的、基于证据的书面审查，并含有指向轨迹细节的超链接，便于开发者定位问题。
- **并排比较**：允许细粒度对比两个智能体在特定维度上的差异，支持回归检测和模型选择。
- **开放源代码**：提供开源基准代码，支持扩展（添加新任务、新 verifier、新裁判维度）。
- **多维度综合指标**：五个 ILM 裁判维度加形式验证，覆盖质量的主要方面，且通过去相关性分析证实其互补性。
- **生产集成**：设计可直接嵌入 CI/CD 管道，支持自动回归检测，实用性强。
- **用户模拟多样性**：通过模拟不同人格的用户（默认、帮助、有毒），评估智能体在不同反馈条件下的鲁棒性。
- **偏差检验**：明确报告并分析位置偏差、自我偏好，并进行了重复运行方差分析，展现了实验的严谨性。

## 8. 不足与局限：包括实验覆盖、偏差风险、应用限制等

- **任务领域局限**：目前仅支持 Java 编码任务，覆盖范围有限。
- **LLM 裁判可靠性**：未正式验证 LLM 裁判与人类评估者的一致性，仅基于内部小型实验。已观察到 LLM 裁判的自我偏好（尤其在“愉悦度”指标上），可能影响排序公平性。
- **外部服务不确定性**：部分模型通过第三方 API 访问（如 OpenRouter），受提供商路由、版本、延迟影响，结果可能不可复现（如 Kimi K2.6 的工具解析 bug 属于提供商问题）。
- **成本较高**：一次完整基准运行（特别是使用前沿模型）可能花费超过 $100，限制了大规模使用。
- **潜在偏向性**：基准最初的开发和优化基于作者自己的智能体系统（Explyt AI Agent），可能导致任务或集成假设更有利于自身系统。作者虽承认并计划通过开源和外部使用来缓解，但中立性仍需验证。
- **形式验证的局限性**：二进制验证本身也可能被“奖励黑客”（reward hacking），尽管轨迹审查可以缓解，但验证器本身可能不全面。
- **用户模拟的局限性**：模拟用户虽然多样，但仍然是模拟，无法完全反映真实用户的复杂性和不可预测性。
