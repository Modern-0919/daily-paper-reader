## 摘要
编码代理几乎完全根据解析率进行排名：即其最终补丁是否通过目标测试。然而，两个代理可以通过完全不同的过程达到相同的结果，单个通过/失败标签无法说明运行为何失败，或为何通过的运行花费了额外的步骤、时间或令牌。这个过程证据存在于轨迹中，轨迹记录了运行的搜索、读取、编辑、工具调用、验证和回退。然而，原始轨迹是异构的，难以在运行之间进行比较。我们提出了TraceProbe，一个轨迹诊断框架，揭示了解析率所掩盖的内容。TraceProbe将每次原始运行标准化为具有确定性效果标签的规范九类动作分类法，然后应用两个基于规则的模块：Insight从已建立的调试实践中识别单轨迹反模式（例如搜索循环、验证跳过），而Converge对齐运行对并分类它们在受控参考下行为发散的位置。将TraceProbe应用于SWE-Bench Verified上五个生产环境中的2500个轨迹，我们发现：(i) 文件选择过于粗糙，无法区分成功与失败，而函数选择和完成行为则能定位；(ii) Insight反模式主要作为语料库级别的难度线索，搜索循环最为稳定；(iii) 即使已解决的运行，在达到相关代码的速度和产生的失败工作量上也存在差异。因此，轨迹结构通过定位检查目标、提出失败假设和优先审查运行，为结果增加了可审计的诊断上下文。

## Abstract
Coding agents are ranked almost entirely by resolve rate: whether their final patch passes the target tests. Yet two agents can reach the same outcome through very different processes, and a single pass/fail label says nothing about why a run failed or why an accepted run spent extra steps, time, or tokens. This process evidence lives in the trajectory, which records a run's searches, reads, edits, tool calls, validation, and reversions. However, raw traces are heterogeneous and hard to compare across runs. We present TraceProbe, a trajectory-diagnostic framework that recovers what resolve rate hides. TraceProbe normalizes each raw run into a canonical nine-type action taxonomy with deterministic effect labels, then applies two rule-based modules: Insight names single-trajectory anti-patterns adapted from established debugging practice (e.g., search loops, verification skips), while Converge aligns pairs of runs and classifies where their behavior diverges under controlled references. Applying TraceProbe to 2,500 trajectories from five production settings on SWE-Bench Verified, we find that (i) file choice is too coarse to separate success from failure, whereas function selection and completion behavior localize it; (ii) Insight anti-patterns act mainly as corpus-level difficulty clues, with search loops the most stable; and (iii) even resolved runs differ in how quickly they reach relevant code and how much failed work they incur. Trajectory structure thus adds auditable diagnostic context to outcomes by localizing inspection targets, suggesting failure hypotheses, and prioritizing runs for review.

## 2. 方法论：TraceProbe

### 核心思想
- 将原始轨迹归一化为 **规范动作序列**（九种动作类型，带确定性效果标签），然后应用两个基于规则的分析模块（Insight 和 Converge），实现单轨迹诊断和跨轨迹比较。

### 关键技术细节

1. **共享表示构建**：
   - **规范动作**：九种类型：FILE_READ, FILE_WRITE, SEARCH, COMMAND, PLAN, NAVIGATE, FETCH, AGENT_SPAWN, REASON。
   - **效果标签**：确定性，基于观测状态转移：SURVIVED（存活）、FAILED（失败）、REVERTED（回退）、JUSTIFIED（合理）、OFF-ANCHOR（偏离锚点）、REASONING（推理）等。
   - **工作流阶段**：UNDERSTAND, PLAN, IMPLEMENT, DEBUG, VALIDATE, REPORT（可选 LLM 语义标签）。
   - **锚点集**：当有基准真值补丁时，使用其修改文件作为锚点；否则通过幸存写入和引用推断。

2. **Insight 模块（单轨迹诊断）**：
   - 定义一系列**反模式探测器**，每个探测器是确定性的谓词，基于固定阈值。
   - 主要反模式：搜索循环（Search Loop）、重读搅动（Re-read Churn）、工具振荡（Tool Oscillation）、无正式尾部验证（No Formal Tail Validation）、未支持的完成声明（Unsupported Completion Claim）等。
   - 此外还有基于 LLM 语义标签的探测器（如阶段振荡、语义无果探索），仅用于探索分析。

3. **Converge 模块（跨轨迹诊断）**：
   - 将一条比较轨迹与一条参考轨迹对齐（基于最长公共子序列 LCS，动作类型、目标、效果标签兼容则匹配）。
   - 识别**发散区间**（未匹配或重排序的动作连续段），并分类为三个诊断层：
     - 文件选择（off-anchor 探索、范围漂移）
     - 编辑稳定性（快速重写、错误螺旋）
     - 完成行为（死胡同、顺序低效）
   - 输出强/中/弱分类，分类阈值固定，分类相对于所选参考。

4. **里程碑（Milestones）**：
   - 基于锚点集提取五个任务相关进度时刻：M1（首次读取锚点文件）、M2（首次写入锚点文件）、M3（所有锚点文件写入）、M4（首次通过验证命令）、M5（首次合理动作）。

## 6. 主要结论与发现

1. **文件选择过于粗糙**：成功/失败轨迹在文件级别可重叠，但**函数选择和完成行为**能更好定位失败证据。
2. **反模式主要作为难度线索**：搜索循环是最稳定的失败关联信号，但其他反模式（如重读搅动）可受难度和设置影响，并非唯一原因。
3. **成功轨迹间仍有过程差异**：在达到相关代码的速度、失败工作量（失败/回退动作比例）上存在明显差异，解析率无法体现。
4. **确定性效果标签揭示过程轮廓**：不同设置（如 Codex 高生存动作、OpenCode 高失败动作）在效果分布上有显著差异，超越原始遥测。
5. **跨基准可转移性有限**：搜索循环方向一致（失败中更常见），但其他探测器关联强度变化，需要在新基准上审计阈值。
6. **Converge 参考依赖性**：文件选择分类对参考变化最鲁棒（0.69% 改变），编辑稳定性较敏感，完成行为居中。
