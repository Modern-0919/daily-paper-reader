## 摘要
工具增强型语言模型代理通常根据是否选择了正确的工具、生成了有效的API参数以及完成了请求的任务来评估。然而，代理可能选择了正确的工具，但仍然作用在错误的外部实体上。例如，请求“给Alex发邮件关于发布事宜”可能导致代理联系错误的Alex、附上错误的发布文档、在错误的邮件线程中回复或更新错误的客户账户。我们将这些错误称为实体绑定失败。本文研究了工具增强型代理中作为一个独特的可靠性和安全性问题的实体绑定失败。我们形式化了工具正确性和实体正确性之间的区别，引入了企业工作流中错误实体失败的分类法，并评估了实体感知的执行机制，包括实体解析前提条件、置信门控绑定、歧义下的澄清以及来源追踪。在跨越60个任务、五个模型后端和六种工具使用方法的受控诊断评估中，所有方法都实现了0.0%的错误工具错误，但面向动作的基线仍会在24.0%-26.0%的运行中产生错误实体的动作。实体感知方法在这种设置下消除了错误实体动作和风险加权的错误实体暴露，但通过延迟处理歧义降低了直接任务完成率。这些发现表明，安全使用工具不仅需要选择正确的工具，还需要在行动之前可靠地将自然语言引用绑定到正确的现实世界实体上。

## Abstract
Tool-augmented language-model agents are often evaluated by whether they select the correct tool, produce valid API arguments, and complete the requested task. However, an agent may choose the right tool and still act on the wrong external entity. For example, a request to "email Alex about the launch" may lead the agent to contact the wrong Alex, attach the wrong launch document, reply in the wrong thread, or update the wrong customer account. We call these errors entity binding failures. This paper studies entity binding failures as a distinct reliability and safety problem in tool-augmented agents. We formalize the separation between tool correctness and entity correctness, introduce a taxonomy of wrong-entity failures in enterprise workflows, and evaluate entity-aware execution mechanisms including entity-resolution preconditions, confidence-gated binding, clarification under ambiguity, and provenance tracking. In a controlled diagnostic evaluation across 60 tasks, five model backends, and six tool-use methods, all methods achieved 0.0 percent wrong-tool error, yet action-oriented baselines still produced wrong-entity actions in 24.0-26.0 percent of runs. Entity-aware methods eliminated wrong-entity actions and risk-weighted wrong-entity exposure in this setting, but reduced direct task completion by deferring under ambiguity. These findings show that safe tool use requires not only selecting the correct tool, but also reliably binding natural-language references to the correct real-world entity before action.

# 论文总结：Entity Binding Failures in Tool-Augmented Agents（工具增强型代理中的实体绑定失败）

## 1. 论文的核心问题与整体含义（研究动机和背景）

- **核心问题**：现有工具增强型语言模型代理的评估主要关注工具选择正确性、API参数有效性和任务完成率，但忽略了一种常见且严重的失败模式——代理虽然选对了工具，却作用在了错误的现实世界实体上。例如，指令“给Alex发邮件关于发布事宜”可能导致代理联系错误的Alex、附加错误的发布文档、回复到错误的线程或更新错误的客户账户。作者将此类错误定义为**实体绑定失败（Entity Binding Failures）**。
- **研究动机**：实体绑定失败在企业场景中尤其危险，因为错误的操作（如发送信息给错误的人、编辑错误的文件、删除错误的记录）可能带来隐私、运营、财务和声誉损害。现有基准（如API-Bank、ToolLLM、τ-bench）主要衡量工具使用能力，但未区分工具正确性与实体正确性。
- **整体含义**：安全可靠的工具使用不仅需要选对工具类型，更需要在执行动作前将自然语言引用可靠地绑定到正确的现实世界实体。本文首次将实体绑定失败作为第一类可靠性和安全性问题进行研究。

## 2. 论文提出的方法论：核心思想、关键技术细节

- **核心思想**：在工具选择和工具执行之间插入一个**实体感知动作门（Entity-Aware Action Gate）**，强制要求在动作执行前完成实体解析，否则进行澄清或推迟。
- **关键技术细节**：
  - **形式化定义**：
    - 工具正确性：`ToolCorrect(a) = 1[t(a) = t⋆]`
    - 实体正确性：`EntityCorrect(a) = 1[ê(a) = e⋆]`
    - 实体绑定失败：`EntityBindingFailure(a) = 1[t(a)=t⋆ ∧ ê(a)≠e⋆]`
    - 多实体任务要求所有绑定均正确。
    - 歧义定义：当候选实体数 > 1时视为歧义，真正歧义任务下执行任何具体动作均不安全。
  - **算法流程（Algorithm 1）**：
    1. 根据用户指令和环境状态选择候选工具。
    2. 获取该工具的实体前提条件（如发送邮件需要收件人实体）。
    3. 提取指令中的实体提及。
    4. 对每个提及，检索候选实体集合，计算绑定分数（置信度）。
    5. 检查绑定是否满足绝对置信度阈值τ和与第二候选的间隔阈值δ，若满足则记录溯源证据并继续，否则要求澄清并返回推迟。
    6. 若所有强制前提条件满足且所有必要实体绑定已解析，则执行工具调用；否则请求缺失信息。
  - **关键机制**：
    - **实体解析前提（Entity-Resolution Preconditions）**：每个工具关联强制/可选实体类型列表，必须在执行前满足。
    - **置信门控绑定（Confidence-Gated Binding）**：绑定仅当最高分≥τ且与第二高分差距≥δ时才视为已解析。
    - **歧义下的澄清（Clarification under Ambiguity）**：当多个候选实体都合理时，生成包含具体元数据的澄清问题，避免模糊提问。
    - **溯源追踪（Provenance Tracking）**：记录每个绑定所依据的证据（如实体ID、名称、邮箱、时间戳等），支持审计和更好的澄清。
  - **实体感知因果最小工具过滤（Entity-Aware CMTF）**：传统CMTF仅基于因果相关性和状态示决策工具可见性，本文扩展为要求实体准备就绪（EntityReady），即只有实体绑定已解析或可路由至澄清步骤的工具才可暴露。

## 3. 实验设计：数据集/场景、基准、对比方法

- **诊断测试集**：自构建的**60个诊断任务**，覆盖五类企业域：Email、Calendar、Documents、Customer Records、Issue Tracking。每个任务包含用户指令、环境状态、工具集、候选实体、任务标注（正确工具、目标实体绑定、期望安全行为）。任务按歧义类型分类：Unambiguous、Name Collision、Document-version Ambiguity、Temporal Ambiguity、Account Collision、Near-duplicate Entity、Cross-system Ambiguity、True Ambiguity。
- **动作风险等级**：Low（读/检索）、Medium（草稿/准备）、High（发送/共享/更新）、Critical（删除/取消/关闭），用于风险加权指标。
- **对比方法**（共6种）：
  1. **Direct**：直接生成工具调用。
  2. **Semantic Filter**：工具按语义相关性过滤后动作。
  3. **CMTF only**：因果/状态依赖工具过滤，无实体门控。
  4. **Entity Retrieval**：提供检索出的候选实体供模型选择。
  5. **Confidence Gate**（实体感知）：仅在实体置信足够时执行，否则推迟/澄清。
  6. **Entity CMTF + Provenance**（实体感知）：工具可见性和执行均依赖于实体前提和溯源证据。
- **模型后端**（5种）：Amazon Nova 2 Lite、Amazon Nova Premier、Claude Opus、Claude Sonnet、Llama 3.3 70B Instruct。
- **总运行数**：5模型 × 6方法 × 60任务 = 1,800次。

## 6. 论文的主要结论与发现

- **核心发现1**：在诊断测试中，所有方法均达到0.0%错误工具率，但面向动作的基线方法（Direct、Semantic Filter、CMTF only、Entity Retrieval）仍产生24.0%-26.0%的错误实体动作。这表明**工具正确性 ≠ 实体正确性**，现有评估指标严重忽略后者。
- **核心发现2**：检索和工具过滤本身并不能可靠防止实体绑定失败（实体检索方法错误实体率26.0%，CMTF only为25.7%，Semantic Filter为24.0%）。
- **核心发现3**：实体感知方法（Confidence Gate和Entity CMTF+Provenance）**完全消除了错误实体动作**（0.0%），并将风险加权错误实体暴露降至0.0，但代价是直接任务完成率从74%–75%降至26%–32%。
- **安全-完成权衡**：实体感知方法通过推迟歧义任务实现零错误实体，而面向动作方法通过猜测获得更高完成率但伴随危险。在风险动作上，澄清比错误执行更可取。
- **歧义类型影响**：时间歧义和真正歧义任务最危险（错误实体率可达90%-100%），名称冲突和跨系统引用次之。实体感知方法对所有歧义类型均能正确检测并延迟行动，且过度澄清率为0.0%（不对明确任务提问）。
- **模型普遍性**：错误实体失败跨越所有模型家族，说明这是一个普遍问题而非特定模型弱点。
