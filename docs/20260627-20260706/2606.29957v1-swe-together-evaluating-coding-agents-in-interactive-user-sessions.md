## 摘要
大多数编码代理基准测试是静态的：代理预先收到完整的任务描述，并仅根据其最终代码进行评判。真正的编码辅助是交互式的，用户会在多轮对话中澄清目标、添加约束并纠正错误。我们提出了SWE-Together，一个从真实用户-代理编码会话中重构的多轮基准测试。为了使真实交互可验证，我们从11,260个记录的会话中精选了109个仓库级任务，选择了那些具有可恢复仓库状态、清晰用户目标和可观测结果的会话。为了跨代理重放这些交互，我们构建了一个基于LLM的反应式用户模拟器，该模拟器保留原始用户的意图，并在编码代理的进展需要时提供反馈。为了评估作为协作者的代理，我们同时衡量最终仓库的正确性以及交互期间所需的纠正反馈轮数。与前沿编码代理的实验表明，更强的代理通常能达到更高的最终成功率，同时需要更少的干预，这表明用户体验得到了改善。

## Abstract
Most coding-agent benchmarks are static: an agent receives a complete task description up front and is judged only by its final code. Real coding assistance is interactive, with users clarifying goals, adding constraints, and correcting mistakes over multiple turns. We introduce SWE-Together, a multi-turn benchmark reconstructed from real user-agent coding sessions. To make real interactions verifiable, we curate 109 repository-level tasks from 11,260 recorded sessions, selecting sessions with recoverable repository states, clear user goals, and observable outcomes. To replay these interactions across agents, we build a reactive LLM-based user simulator that preserves the original users' intents and provides feedback when the coding agent's progress requires it. To evaluate agents as collaborators, we measure both final repository correctness and the number of corrective feedback turns required during the interaction. Experiments with frontier coding agents show that stronger agents generally achieve higher final success rates while requiring fewer interventions, suggesting an improved user experience.

## 1. 核心问题与整体含义（研究动机和背景）

- **现有基准的静态性与现实需求的矛盾**：当前主流编码代理基准（如 SWE-Bench 系列、Terminal-Bench）都采用静态协议：预先给出完整的任务描述，仅通过最终代码执行测试来评分。这种设置忽略了真实编码辅助的核心特性——交互性。在实际使用中，用户往往通过多轮对话逐步澄清目标、添加约束、纠正错误，而非一次性给出完整指令。
- **评估维度的缺失**：现有基准仅衡量最终任务成功率，无法体现代理吸收演化指令的能力以及用户需要付出的纠正努力。两个代理可能获得相同的最终分数，但一个能从粗略的初始请求成功，另一个则需要详细的说明、反复提醒和大量纠正反馈。
- **SWE-Together 的目标**：填补交互式评估的空白，构建一个从真实用户-代理会话中重构的多轮基准，既评估最终仓库正确性，也量化交互过程中的用户引导成本，从而更贴近实际用户体验。

## 2. 方法论：核心思想、关键技术细节、算法流程

- **核心思想**：将真实的多轮编码会话转化为可验证、可重放的仓库级任务，并通过一个锚定于原始会话的 LLM 用户模拟器来重放交互，确保不同代理的评估可比。
- **关键技术细节**：
  - **会话到任务的构建流程**：
    1. 确定性过滤：从 4 个上游数据源（DataClaw、Pi-staging、Hyperswitch、SWE-chat，共 11,260 条会话）中，筛选出具有多轮用户交互、具体代码修改、可识别仓库的会话，排除主要由用户编写最终变更的会话。
    2. 可行性筛选：使用 LLM 判断会话的编码工作是否能在本地沙盒中重现，排除依赖外部状态（如 PR 管理、部署、私有凭据等）的任务。
    3. 任务生成：在隔离沙盒中，由任务生成代理克隆仓库到固定提交、识别设置和测试命令、编写可执行验证、用户模拟提示等，生成完整的任务包。
  - **用户模拟器**：
    - 基于 LLM，采用**状态条件决策策略**：每次代理完成一轮后，模拟器根据固定锚点（原始会话分析）、代理最新轨迹、自身记忆等，决定发送何种消息（no-op/clarification/redirect/new-requirement/check-external）或不发言。
    - 锚定于原始会话：任务包中包含从原始对话分解出的用户意图、触发条件，使模拟器的反馈与原始用户行为一致，避免固定重放或漂移。
  - **评估方法**：联合评估两个维度：
    - **任务正确性**：使用一个两阶段的**代理化评分器**：
      - 阶段1：离线为每个任务生成加权评分量表（基于原始参考补丁和仓库检查），定义一组行为目标及其权重。
      - 阶段2：将同一评分量表应用于每个候选仓库状态，返回每个目标是否达成（0/1），最终分数为加权和并四舍五入到两位小数：`score = round(∑_g w_g · I[g met], 2)`。
    - **用户模拟行为指标**：
      - **意图覆盖率**：衡量重放消息忠实保留原始用户意图的程度，结合加权意图召回（0.7）和范围精度（0.3）计算。
      - **用户纠正**：量化代理诱发的纠正性引导，每次显式纠正计1分，每次轻微“轻推”计0.2分，然后按任务和副本平均。

## 3. 实验设计

- **数据集/场景**：
  - 使用 SWE-Together 基准本身，包含 109 个从真实会话构建的仓库级任务。任务覆盖多个 GitHub 项目，环境已固定。
- **Benchmark 对比**：
  - 对比了 7 个前沿模型：Claude Opus 4.8、GPT-5.5、Claude Opus 4.6、GLM-5.2、GLM-5.1、DeepSeek-V4-Pro、MiniMax-2.7。
  - 所有模型使用统一的代理框架 `opencode`，每个任务进行 2 次重复（k=2）。
  - 参考基准（Oracle）：使用原始会话的参考补丁在相同评分标准下得分。
- **评估指标**：
  - 正确性指标：pass@1（单次通过率）、SSR（稳定通过率，先平均连续评分再阈值化）、pass@2（两次均通过的比率）、MeanJudge（平均连续评分）。阈值 τ=0.85。

  - 效率指标：每任务输出+推理令牌数、墙钟时间。

## 6. 主要结论与发现

- **正确性排序**：Claude Opus 4.8 在所有正确性指标上领先（pass@1=63%，MeanJudge=0.801），GPT-5.5 第二，MiniMax-2.7 最差。所有模型远低于 Oracle 参考补丁的 78% 通过率和 0.90 平均分，说明交互式评估具有挑战性。
- **用户纠正与能力负相关**：用户纠正与 pass@1 的 Pearson 相关系数为 -0.92，与稳定通过率为 -0.84。更强模型所需的纠正反馈显著更少（Claude Opus 4.8 平均 1.38，MiniMax-2.7 为 2.17），且该关系在硬子集上依然成立。
- **模拟器一致性**：意图覆盖率在 6 个模型上基本稳定（0.70-0.72），GPT-5.5 略低（0.68），但仍在合理范围内，说明模拟器行为在不同代理间具有可比性。
- **模拟器质量**：人类注释者无法可靠区分模拟用户与真实用户（Turing 通过率 46%，95% CI 含 50%），表明模拟器具有高度真实性。
- **效率差异**：GPT-5.5 在正确性排名第二的同时，每任务令牌数（29.9k）和墙钟时间（10.7 分钟）最低，展现了最佳效率平衡；Claude Opus 4.8 在正确性最佳但令牌成本最高（74.0k）。

## 7. 优点（方法或实验设计上的亮点）

1. **真实数据驱动**：任务来源于 11,260 条真实用户-代理会话，而非人工编写或从静态问题合成，生态效度高。
2. **可重放的多轮交互**：创新的“锚定+状态条件”用户模拟器，既能适应不同代理的轨迹，又能保持原始用户意图，解决了可验证与真实性的矛盾。
3. **联合评估框架**：同时考虑最终正确性和交互成本（用户纠正），提供更全面的代理协作能力画像，弥补了传统静态基准的缺陷。
4. **评分器设计合理**：使用离线生成的固定评分量表，避免了对特定方案的偏见；区分行为目标而非代码相似性，更具鲁棒性。
5. **模拟器审计**：引入意图覆盖率和人工 Turing 测试来验证模拟器质量，确保评估的可靠性和可比性。
6. **开源与可复现**：代码和网站已公开，任务环境固定，便于社区复现和扩展。

## 8. 不足与局限

1. **基准规模较小**：仅 109 个任务，与 SWE-Bench 等大规模基准相比，统计分析可能不够稳健；部分任务可能缺少多样性。
2. **模拟器能力受限**：用户模拟器不能在代理执行期间中断干预，不能直接编辑文件，仅依赖文本轨迹（无视觉界面信息），可能遗漏某些真实交互模式（如用户直接修改代码）。
3. **任务类型偏窄**：专注于有明确、可测交付物的任务（如提交补丁），对模糊、开放性的编码任务（如探索性设计、代码重构）覆盖不足。
4. **参考基线不是完美天花板**：参考补丁的平均分仅 0.90，通过率 78%，原因包括过程性要求、提取噪声等，使得“天花板”并非 1.0，可能限制对顶尖能力的区分度。
5. **计算资源未报告**：未提供实验所需的 GPU 型号、数量及时间，影响可复现性和成本估算。
6. **重复次数有限**：每个任务仅 2 次重复，无法充分估计运行间方差；pass@2 等一致性指标可能不稳定。
7. **未评估不同代理框架的差异**：所有模型使用同一 `opencode` 框架，未测试框架本身对结果的影响。
8. **模拟器意图覆盖率仍有下降空间**：GPT-5.5 的意图覆盖率略降，提示能力更强的代理可能减少模拟器自然发言的机会，可能导致信息遗漏的偏差。
