## 摘要
向完全自主的电信网络（自主网络级别4-5）的演进要求AI/ML代理在没有人工干预的情况下做出实时网络决策。然而，目前缺乏标准化的运行时机制来在单个推理输出触发实时网络状态变更之前进行拦截和验证，这带来了错误自主决策的风险。本文提出了防护栏验证（GRV）框架，一种标准化的运行时架构，用于在执行前拦截和验证AI驱动的决策。该框架通过多个加权维度（包括行动范围、行动类型、服务关键性、代理自主级别、可逆性和时间行为模式）来评估决策，确定关键性级别。基于该级别，应用渐进的验证机制：执行并记录、边界检查、独立代理验证或多代理共识。该框架还提供跨代理冲突检测，采用关键性加权的优先级解决，以及运行时合规性日志记录，以满足法规要求（例如欧盟AI法案第14条）。我们展示了架构、算法流程、O-RAN部署模型，并评估了针对电信领域中已知AI/ML攻击的威胁覆盖范围。

## Abstract
The evolution toward fully autonomous telecommunications networks (Autonomous Network Levels 4-5) requires AI/ML agents to make real-time network decisions without human intervention. However, no standardized runtime mechanism exists to intercept and validate individual inference outputs before they trigger live network state changes, creating risks of erroneous autonomous decisions. This paper proposes the Guard Rail Validation (GRV) framework, a standardizable runtime architecture for intercepting and validating AI-driven decisions before execution. The framework evaluates decisions across multiple weighted dimensions -- including action scope, action type, service criticality, agent autonomy level, reversibility, and temporal behavioural patterns -- to determine a criticality level. Based on this level, graduated validation mechanisms are applied: execute-with-logging, bounds checking, independent agent validation, or multi-agent consensus. The framework additionally provides cross-agent conflict detection with criticality-weighted priority resolution and runtime conformance logging for regulatory compliance (e.g., EU AI Act Article 14). We present the architecture, algorithmic procedures, O-RAN deployment model, and evaluate threat coverage against known AI/ML attacks in telecommunications.

- **核心问题**：在向全自主电信网络（L4-L5）演进过程中，AI/ML代理实时生成网络决策，但现有3GPP/O-RAN标准缺乏**运行时验证机制**——推理输出一旦产生，直接触发网络状态变更，无安全拦截。这导致五个关键缺口：
  1. 无预执行验证：部署后无运行时保护。

  4. 无冲突解决：多代理环境中可能产生矛盾决策，缺乏检测与协调。
  5. 无法规合规：不能满足EU AI Act等对高风险AI系统的人为监督要求。
- **研究含义**：提出**Guard Rail Validation (GRV)**框架，在推理输出与网络执行之间插入标准化的验证层，按决策风险级别采取不同验证措施，同时提供跨代理冲突检测和合规日志，以支撑自主网络的可靠运行。

## 2. 论文提出的方法论

### 核心思想
GRV框架作为强制验证层，拦截每个推理输出，经过四个子流程后决定执行/阻止/升级：**关键性评估 → 渐进验证 → 冲突解决 → 合规日志**。验证机制与决策风险成正比，低风险仅记录，中风险边界检查，高风险独立代理验证，极高风险多代理共识。

### 关键技术细节
- **关键性评估**：基于六个加权维度计算总得分（满分21分），映射为LOW/MEDIUM/HIGH/CRITICAL四个等级：
  - 行动范围（单元素=1～全网=4）
  - 行动类型（读=0～关闭=4，由模式自动分类，防止恶意伪装）
  - 服务关键性（mMTC=1～Emergency=4）
  - 自主级别（建议=0～自动=2）
  - 可逆性（即时=0～不可逆=2）
  - 时间模式（正常=0～振荡/突发=2，通过滑动窗口检测频繁、振荡或突发行为）
- **渐进验证**：
  - LOW（0-4分）：仅执行并记录日志。
  - MEDIUM（5-8分）：边界检查（参数范围、变化率、硬约束）。
  - HIGH（9-12分）：独立验证器（另一架构/数据的ML模型或规则）返回同意/不同意/不确定；不确定时按降级策略执行。
  - CRITICAL（13+分）：M-of-N共识（默认3个验证器，2/3同意，Emergency切片需全票）；超时或验证器不足时降级。
- **冲突检测与解决**：短时决策队列中检查同一实体/参数的矛盾意图，按优先级（切片关键性 > 动作紧急度 > 代理可信度 > 运营规则 > 先到先得）裁决。
- **合规日志**：结构化记录决策全链路（代理、动作、关键性分数、验证结果、冲突处理、最终结果），满足EU AI Act透明度要求。
- **O-RAN集成**：Non-RT RIC/SMO运行完整GRV；Near-RT RIC采用轻量变体（仅边界检查+异步时间监控+升级到Non-RT RIC）。

### 算法流程（文字说明）
- **时间模式检测算法**：按(代理, 参数, 目标)键维护滑动窗口；若同类决策数超过频率阈值则标记频繁；若窗口内存在反向操作则标记振荡；若不同代理同一目标数超过突发阈值则标记突发。
