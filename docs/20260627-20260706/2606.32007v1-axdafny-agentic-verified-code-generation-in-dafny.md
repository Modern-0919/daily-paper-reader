## 摘要
我们研究Dafny中的智能体代码生成，其中模型必须生成可执行代码和用于验证的证明制品。我们提出了AxDafny，一个验证器引导的修复框架，能够迭代生成实现、不变量、断言和终止论证。我们还引入了LiveCodeBench-Pro-Dafny（LCB-Pro-Dafny），这是一个包含250道竞赛风格编程问题的基准测试，这些问题被翻译成带有形式规范的Dafny代码，并配备了基于验证器的评估工具。在LCB-Pro-Dafny上，AxDafny显著提高了基线GPT-5.5性能的验证成功率。在DafnyBench上，AxDafny达到了92.7%的验证成功率，比先前报道的最强证明提示基线高出6.5个百分点。最后，我们展示了验证成功率和运行时测试性能衡量的是生成代码的不同方面。

## Abstract
We study agentic code generation in Dafny, where a model must generate both executable code and the proof artifacts for verification. We present AxDafny, a verifier-guided repair framework that iteratively generates implementations, invariants, assertions, and termination arguments. We also introduce LiveCodeBench-Pro-Dafny (LCB-Pro-Dafny), a benchmark of 250 competition-style programming problems translated into Dafny with formal specifications and a verifier-based evaluation harness. On LCB-Pro-Dafny, AxDafny substantially improves verification success over baseline GPT-5.5 performance. On DafnyBench, AxDafny achieves 92.7\% verification success, outperforming the strongest previously reported proof-hint baseline by 6.5 percentage points. Lastly, we show that verification success and runtime test performance measure different aspects of generated code.

- **背景**：大语言模型在代码生成中常采用“生成-检查-修复”的迭代循环，而形式化验证技术能提供比固定测试用例更丰富的反馈信号（如违反规范、缺少证明注解、未处理边界情况等）。然而，形式化方法在主流AI任务中的应用受限于其巨大的专业门槛。
- **问题**：现有代码生成模型难以同时保证代码的功能正确性和形式化验证，尤其对于需要同时生成可执行代码和证明注解（如不变量、断言、终止论证）的验证语言（如Dafny）。Dafny是一种内建形式化验证支持的语言，但其训练数据稀缺（Python文件数量比Dafny文件多三个数量级）。
- **整体含义**：研究验证器引导的智能体代码生成，旨在提升生成代码的可信度，同时探索验证成功与运行时测试之间的差异。

## 2. 方法论：核心思想、关键技术细节
- **核心思想**：提出AxDafny框架，采用“验证器引导的修复”（verifier-guided repair）循环，迭代生成Dafny代码和证明注解，最终通过Dafny验证器检查。
- **技术细节**：
  - **智能体架构**：由提议者（Proposer）、审查系统（Reviewer System）和内存模块（Memory Module）组成。
    - **提议者**：接收原始问题、Dafny规范、历史上下文，生成完整的Dafny文件（包括实现和注解），目标是通过`dafny verify`且不修改给定的`requires`/`ensures`。
    - **审查系统**：两阶段检查。
      - **确定性检查**：① 验证`requires`/`ensures`子句未削弱（原始子句是提议子句的子集）；② 正则过滤禁用`{:axiom}`、`{:verify false}`、`assume`、`{:extern}`等绕过验证的构造。若不通过，返回违规行给提议者。
      - **Dafny验证器**：运行`dafny verify`，获取失败证明义务的诊断信息（包括源代码位置和反例，若启用`--counterexamples`），反馈给提议者。
      - **LLM审查**：额外检查提议文件是否存在作弊模式（如重写谓词使其平凡为真）。仅当所有检查通过且验证成功时接受候选解。
    - **上下文管理**：采用“自管理反射内存”（self-managed reflective memory），每次迭代提供给提议者：① 上一次尝试的完整内容（推理、文件、反馈）；② 通过单独调用LLM维护的滚动记忆（rolling memory），压缩所有历史教训并整合新发现。
